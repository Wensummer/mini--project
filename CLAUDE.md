# 基于数据标签挖掘的企业登记注册合规风险监测

竞赛项目。3 人团队。本文件是项目上下文，每次会话先读它。

---

## 一句话定位

**不是做一个更准的风险模型，是做一个更可信的判定机制。**

监管场景里，precision 从 0.91 提到 0.94 没人在乎；"你凭什么这么判"和"判错了怎么办"才是全部。

系统要能自动发现企业登记数据里"这不对劲"，**而且每次说"不对劲"，都能说出是哪条法规的哪一款不对劲**。

---

## 六条设计原则（已定，不要重新讨论）

1. **证据先于结论** —— 判定由规则命中产生，证据链即解释本身。不做 post-hoc rationalization（先出分数再让 LLM 编解释）。
2. **LLM 严格旁路** —— 大模型不参与任何判定，只做翻译、检索、排版。主链必须是确定性规则引擎。
3. **规则即数据** —— 规则存库，引擎只做解释器，改规则不发版。
4. **图谱先于标签** —— 关系类标签（同址聚集、股权环、跨省任职）必须先有图谱才算得出。顺序不能反。
5. **只读标签库** —— 下游应用只读标签库，不直接读原始数据。保证三级分析口径一致。
6. **监测已发生 ≠ 预测未来** —— 识别已发生的行为序列，不预测谁"将要"违法。后者法律上答不出。

## 系统管线

```
输入（登记数据 / 法规文本 / 外部关联数据）
  → 数据归集治理（标准化 · 清洗 · 实体对齐 · 主数据）
  → 知识层（业务知识图谱 | 法规向量库 | 规则库）
  → 计算层（规则引擎 | 图算法 | 图约束冲突检测）
  → 标签库  ← 全系统的"腰"，唯一数据出口
  → 应用层（三级分析引擎 | 智能助手）
  → 输出（风险预警工单 | 质量问题清单 | 态势研判报告）
  → 人工复核结果回流 → 规则调优 ↻
```

最后那个回流箭头是方案的灵魂。没有它只是报警器，有了它才是自我进化的规则治理系统。

## 已核实的核心结论（可直接用，不必再查）

### 假说（全项目最有力的论证）

> **统一接口覆盖到哪里，标准就统一到哪里。没有接口的地方，各地各显神通。**

| 校验事项 | 总局统一接口 | 9 省实际 | 结果 |
|---|---|---|---|
| 人员任职 | **有**（实名核验） | 9/9 实名+电签，5 省明确可免交任职文件 | **趋同** |
| 经营范围 | **有**（规范表述目录） | **9/9 目录勾选** | **趋同** |
| 住所真实性 | **无** | 至少六种机制，无一相同；北京甚至发散到**地址级** | **发散** |
| 注册资本门槛 | **无** | 浙江 1000 万 / 河南 100 万 | **发散** |

两个"有"全部趋同，两个"无"全部发散。论据全来自公开办事指南，可复现。

### 已核实属实
- **浙江**：名称冠"浙江"的企业，注册资本最低认缴 **1000 万**
- **河南**：名称冠"河南"的企业，注册资本一般 **100 万**以上
- 《公司法》认缴制下总局对注册资本**零最低要求** → 纯属地自设门槛，差 10 倍
- 陕西自 2026-05-06 起启用新版材料规范（国市监注发〔2026〕5 号是 5 月 1 日施行 → **5 天时滞**）
- 辽宁经营范围不是随意填写（来源：总局登记注册局）
- **北京住所 = 双轨制**：地址已纳入"标准化地址库"→「零材料」系统点选；未纳入 → 传统上传产权证明。两说不矛盾，是同一机制的两个分支。
  来源：bjrd.gov.cn 2022-06-20《区块链助力"零材料"办企业 北京市海淀区率先实现住所标准化登记》（访问 2026-07-17）
  ★ 该文明确写"**海淀区在全市率先**"，2022-06 覆盖产权方 354 个 / 经营性住所 1291 处 = **区级试点，非全市**。2026 现行覆盖范围**待核**（新闻讲改革亮点，办事指南才讲到底要交什么）。
  → **对假说是加强**：住所的发散不止到省级，而是深入**地址级** —— 同城同街，隔壁楼在库你就零材料，你不在库就交房产证。比浙江1000万 vs 河南100万更狠，后者至少省内统一。

## 第一条链路（✔ 已完成 2026-07-17）

目标不是做完，是**证明这条链能通**。已通，且比原计划多跑了两条规则与整个指标层：

```
schema → 1000 家假数据 → 规则库 → 引擎解释 → 标签+证据 → 复核回流 → 处置自调 ↻
                                              ↘ 指标库 → 宏观趋势（无趋势表）
```

### 三条起始规则（正好覆盖三种规则类型；现已扩到 5 条）

| 规则 | 类型 | 判定逻辑 | 技术 |
|---|---|---|---|
| `R-CZ-01` 出资与注册资本不符 | 法定强制 | `SUM(股东.认缴出资额) <> 企业.注册资本` | 纯 SQL |
| `R-ZS-02` 住所高度聚集 | 统计异常 | `同址主体数 > 全省 P99 AND 30日新增 > 阈值` | 分位数基线 |
| `R-BG-05` 股权自环 | 法定强制 | 图上存在有向环 | 图算法 |

> **R-ZS-02 的时间窗不是可选项**（2026-07-17 实测，1000 家合成数据）：
> 只用 `同址主体数 > P99` → **precision = 0.500**，合法的集群注册地址（江苏政策允许一址多照）被全部误报。
> 加上 `30日新增 > 阈值` → **precision = 1.000**。合法集群是每月涓流（1~2 家），虚假批量是 30 天暴增（50 家），**只有时间窗分得开**。
> 这个缺陷只有造了**难负例**才暴露得出来 —— 只造正例的话两版规则都是 precision 1.0。难负例不是为了让分数难看，是为了让错的规则现原形。
> ★ 那个阈值后来查明**不能用分位数定**（30日新增是零膨胀分布，P99=0）—— 见下方「跑出来的结论」第 4 条。
> 现状：`threshold_calibration` 表记依据，未校准的阈值强制降级为「仅打标」。R-ZS-02 当前即处于此状态。

> **R-BG-05 的法条锚点尚未核实**：方案 §6.3 的 C5 只写了"《公司法》/《受益所有人办法》"，无条款号；
> 而《受益所有人信息管理办法》第 2 条是**备案义务**，不是"股权不得成环"。
> 按本文件规矩，法条锚点如实写「第__条 待核」。要么核出条款号，要么降级为不需要法源的规则类型。

### 约束
- **1000 家，不是 10 万。**先验证链路。
- schema 只取矩阵B 里 9 省全 ✓ 的字段（**北京已补齐，核心口径 = 11 个字段**），属地差异字段先不管。
- 合成数据必须带 **ground truth**（注入异常时记录 entity_id + 违反的 rule_id）——这是相对其他队最大的优势，能报 P/R。
- 每条标签实例必须挂 `evidence_id`，指向：规则版本号 + 字段值快照 + 图谱子图 + 法条锚点。

## 跑出来的结论（比文档硬，因为是实测的）

**1. confidence ≠ precision，混了就出事。**
曾把 confidence 改成"复核回流实测覆盖"，结果 R-CZ-01（等式判定，逻辑上不可能错）
因复核员 4% 误判被拉到 0.933 并自动降级 —— 规则没错，是人错了。
方案 §7.4 早写了"把它当模型输出，就把确定性规则污染成概率了"，一字不差。
→ **confidence** = 这条标签说的是不是事实（类型先验，恒定，法定强制 1.0）
→ **precision** = 命中后是否真构成要处置的问题（复核实测，驱动**处置级别**）
confidence 可由 rule_type 推出，但不是冗余 —— 是反范式物化，下游不该为了知道
一条标签铁不铁而回去 join 规则库。

**2. 冷启动时没有任何规则被允许阻断。**
"实测 precision 未经证明（复核 < 20 条）→ 不允许阻断"是引擎的自然推论，非特意设计。
答辩可用：问"规则误判导致企业开不了业谁担责" → **系统结构上不允许未经证明的规则阻断**。

**3. 金额必须 NUMERIC，绝不能 FLOAT。**
R-CZ-01 是等值比较，浮点误差凭空造假阳性。已用 33.3333+33.3333+33.3334=100.0000 验证。

**4. ★「阈值必须来自数据分位数」这条规矩是错的（已实测推翻）。**
- P99 里的 **99 本身就是拍的** —— 分位数只是把"拍绝对数"换成"拍百分比"；
- 分位数保证**恒定告警率**（P99 永远报 1%），与真实风险无关：全省干净照报 1%（全误报），
  全省烂透也只报 1%（大量漏报）；
- **零膨胀分布上分位数直接退化**：实测「30日新增」P90=P95=P99=**0**（415/419 个地址近 30 天零新增），
  按分位数定阈值 precision 掉到 0.500，反倒是"拍的常数 10"precision=1.000。
  **有依据的错了，没依据的对了。**
- 为什么同址数能用 P99 而 30日新增不能：前者分布展开（1→12 连续），后者 99% 是 0。
  → **分位数只在分布足够展开时成立。**

**4b. 阈值确定分三层，取决于手上有什么**（见 `threshold_calibration` 表）：
| 依据类型 | 方法 | 何时可用 |
|---|---|---|
| **复核数据校准** | 扫描候选值，取满足该处置级别 precision 门槛的最小阈值（recall 最大） | 有回流数据后 |
| **告警负荷倒推** | "全省一天能复核 N 条 → 阈值定在产出 N 条处"。依据是运营的不是统计的，但**能说出口**，且是真实监管系统的做法。方案 §7.6 把告警负荷写成事后"检验"—— **它其实该是阈值的定义者** | 有业务约束时 |
| **未校准初值** | 承认定不出。★ **强制降级为「仅打标」** | 冷启动 |

> **不知道一个阈值准不准，就不配让人为它跑腿（复核要实地核查），更不配拿它阻断企业开办。**
> 这与"冷启动无规则可阻断"是同一条逻辑的两个面。
> 问题由此从"阈值定多少"（答不出）变成"阈值凭什么可信、不可信时怎么办"（答得出）——
> 后者才是监管场景真正要问的，也正是本项目的定位。

**4c. ★ 合成数据定不出阈值（诚实的负面结果）。**
实测扫描：达标区间 **[2, 49]，平台宽 24.5×** —— 数据根本没约束住阈值，
"校准"等于在平台上随便挑点。根因是合成数据在合法集群（近30日新增 1~2 家）
与虚假批量（50 家）之间留了一条**不真实的宽沟**；真实分布是连续的。
`threshold_calibration.plateau_low/high` 把平台宽度记着 —— **平台越宽 = 数据越定不住阈值**。
→ R-XN-01 同病（基线样本量仅 7 个登记机关）。→ 进 §11 负面结果。

**5. 规则能证明差异存在，不能证明差异成因。**
R-CZ-01 命中 = 虚假申报 还是 录入错误？R-XN-01 命中 = 尺度过严 还是 申报质量差？
数据分不出来。→ 坦白这条反而符合"监测已发生 ≠ 预测未来"：
**判定矛盾是客观的，判定造假是要人来做的。**

**6. 命中率检验在合成数据上是循环论证。**
§7.6 说"命中率分布最实用"，但合成数据的命中率完全由注入比例决定 —— 测出来的是自己填的数。
该条检验须真实数据才有效。→ 进 §11 负面结果。

## 对方案的修正（已定，勿回退）

**原则 5 收窄**：原文"下游只读标签库，不直接读原始数据"太绝对，与 §8.1 宏观层
（新设趋势、资本结构演变 —— 都不是任何标签的聚合）直接打架。
根源是 §7.2 定义了"标签/特征/指标"三概念，但 §7.4 的数据模型只实现了标签，指标无处安放。
赛题原文本就并列："构建数据标签**和**分析预警体系"；且宏观是"研判"、中观是"分析"、
只有微观才是"预警" —— 宏观本就不该只看风险。

> **修正后**：任何一个口径只允许被计算一次，结果物化为带版本、带时点的资产；
> 下游只消费资产，不重新计算。口径打架的病根是**重复计算**，不是"读了原始数据"。
> 风险口径 → `TAG_INSTANCE`｜统计口径 → `METRIC_INSTANCE`｜阈值基线 → `STAT_BASELINE`。
> **三者合起来才是唯一出口。**

`METRIC_INSTANCE.source_type`（`tag_agg` / `raw_agg`）让"这个数从标签算还是从原始数据算"
成为显式可审计的事实，而非架构争议。

**补业务流程层**：`application`（业务单据）+ `authority`（登记机关）。
课题介绍原文是"梳理**全流程**数据标签体系"，光挂企业挂不住。
`application.ent_id` **必须可空** —— 事前拦截时统一社会信用代码尚未发放，企业实体不存在。
触发时点分层（事前轻量 / 事后批量）同时是"系统承载不足"的解法。

## 答辩预案（评委必问）

| 问题 | 答案 |
|---|---|
| 误报怎么办？ | 分级处置。阻断级须 precision > 0.99 才允许设为阻断，否则降级复核。误报回流成规则调优的标注数据 |
| 规则更新滞后怎么解决？ | 规则可配置化 + 法规变更触发规则库版本更新 + 误报自动沉淀为目录缺口建议。不是"重训模型" |
| 各地标准不一怎么统一？ | **统一规则中台 + 属地差异化参数**。不说"全国一刀切" |
| 数据是假的，结论有意义吗？ | 数据合成，但规则/流程/法条引用全部真实可溯源。且正因合成才有 ground truth，能报 P/R —— 用真实数据的队反而做不到 |
| 给企业评分合法吗？ | 引《关于推进企业信用风险分类管理的意见》：分类是**监管资源配置的内部参考依据，不是对企业的信用评价** |
| LLM 误判导致企业开不了业谁担责？ | LLM 不参与任何判定 |
| 怎么保证 LLM 的解释是真的？ | explanation-by-construction。**"解释不是 LLM 生成的，解释就是证据链本身，LLM 只做排版。"** |
| 一天报多少条？看得完吗？ | 已做告警负荷估算：日均 X 条 / 复核人力 Y 人 = 人均 Z 条 |
| 和天眼查/企查查区别？ | 商业征信做事后关系挖掘；本系统做事前合规校验 + 数据矛盾检测 + 规则治理闭环，每条结论挂法条锚点 |

## 工作原则

**AI 给的是线索，不是证据。**线索用来决定去查什么，证据才能写进方案。

已验证：AI 整理的省份资料命中率约 6/8 —— 够好，但会错、会自相矛盾。
凡涉及法条、日期、地方政策，一律回官网核，截图存档，记 URL 和访问日期。

**漂亮的结论最该被怀疑。**"陕西商洛反向加码"就是因为太漂亮才是错的。
