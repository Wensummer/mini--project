## 三、已知问题（真实存在，没装作解决）

### 3.1 ✔ 已解决：SCD2 落地，复核历史不再被连坐清空

**原问题**：`engine.py` 每次运行 `TRUNCATE review, tag_instance, evidence, stat_baseline`，
而 `review` 外键引用 `tag_instance` —— 每跑一次规则，复核历史就被连坐洗一次，
实测 precision 永远算不出来，回流闭环半瘫。

**已改为 SCD Type 2，什么都不 TRUNCATE**：
- 仍命中 → 保持 `valid_to IS NULL`，`valid_from` 不动（同一事实的延续，不是新事实）
- 不再命中 → `valid_to = now`（失效，记录永存）
- 新命中 → 插入新实例 + 新证据
- `evidence` 不可变；`stat_baseline` 是历史基线（被 `evidence.baseline_id` 引用，删了就答不出
  「半年前这条为什么没报」）；`review` 增量复核，只处理未复核过的实例

**实测验证**：第 1 轮「新增 30 / 延续 0 / 失效 0」→ 复核 59 条 →
第 2 轮「新增 0 / 延续 30 / 失效 0」，`review` 存活，
`R-CZ-01` 的降级理由从「复核样本仅 0 条」变为「**实测 precision 0.933 未达阻断门槛**」
—— 处置级别第一次由实测数据驱动，回流闭环真正闭合。

**失效路径也已验证**：把 `R-RY-01` 阈值 5→20（纯 UPDATE，`engine.py` 一字未动）
→ 4 条职业法人标签失效，但记录永存：命中日、失效日、当时复核结论「成立」、
当时字段快照「任法代数 12」全在。**这就是「半年前这条为什么没报」的答案。**

### 3.2 ⚠ `R-BG-05` 法条锚点未核实

而《受益所有人信息管理办法》第 2 条是**备案义务**，不是「股权不得成环」。

**三条路**：核出条款号 / 降级为不需要法源的规则类型 / 保持现状并在答辩中主动说明。

### 3.3 ⚠ 合成数据定不出阈值

扫描实测：`R-ZS-02` 的 `new_threshold` 达标区间 **[2, 49]，平台宽 24.5×**。
数据根本没约束住阈值，「校准」等于在平台上随便挑点。

**根因**：造数时在合法集群（近30日新增 1~2 家）与虚假批量（50 家）之间
留了一条**不真实的宽沟**。真实分布是连续的。

`threshold_calibration.plateau_low/high` 把它记着 —— **平台越宽 = 数据越定不住阈值**。
`R-XN-01` 同病（基线样本量仅 7 个登记机关）。

→ **必须进方案 §11「主动报告的负面结果」。**

### 3.4 命中率检验在合成数据上是循环论证

### 3.5 规则能证明差异存在，不能证明差异成因

- `R-XN-01` 命中 = 该机关尺度过严？还是该区申报质量本来就差？

**数据分不出来。** 直接影响「误报」的定义 —— 而「你的误报怎么办」是答辩必问第一题。

→ 建议坦白，且这条**反而符合**原则 6「监测已发生 ≠ 预测未来」：
**判定矛盾是客观的，判定造假是要人来做的。**

### 3.6 ✔ 已解决：「系统承载不足」已成文（方案新增 §6.4）

EXPLAIN 实测：事前拦截 `R-CZ-01-PRE` 走 **Index Scan**，扫 50 条在途，**3 buffers**；
事后回溯 `R-CZ-01` 走 **Seq Scan ×2**，扫 1000 企业 + 1907 出资，**47 buffers**。

关键不是 15.7 倍，是**两条曲线斜率不同**：事前 O(在途量)（稳态），事后 O(存量)（只增不减）。
→ 全放事前 ⇒ 每笔办件触发一次全表扫描 ⇒ 办件延迟随存量线性增长 ⇒ 系统卡死。
→ 分层后办件延迟与存量无关。**不是堆机器，是让 99% 的办件只走轻量路径。**

★ 顺带修掉一个自己引入的错误：`R-CZ-01` 曾照抄方案样例标为「事前拦截 + 硬阻断」，
但其 logic 查 `enterprise`（已登记企业）—— 企业都登记完了，拦什么？阻断什么？
已改为「事后回溯 + 人工复核」。
**新约束：规则声明的触发时点，必须与其 logic 实际作用的对象一致**
（事前→只能查 application；事后→查 enterprise）。违反则承载分层失效 + 处置动作无意义。
当前 6 条规则一致性自检 **6/6 通过**。

### 3.8 ★ 方案 §6.3 的「八条图约束」里，七条根本不需要图

逐条拆 Cypher：

| 约束 | Cypher 的实质 | 真需要图吗 |
|---|---|---|
| C1 出资和≠注册资本 | 单跳 + 聚合 | 否，`GROUP BY HAVING` |
| C2 失信人任职 | 单跳 | 否，`JOIN` |
| C3 许可缺失 | 单跳 + NOT EXISTS | 否 |
| C4 注销后变更 | 单跳 | 否，`WHERE` |
| **C5 股权环** | **`-[:投资*1..6]->` 变长路径** | **是** |
| C6 代理人兼联络员 | 固定 2 跳 | 否，两个 `JOIN` |
| C7 特种设备许可 | 单跳 | 否，`JOIN` |
| C8 一码多名 | **连边都没有**，纯自连接 | 否，`SELF JOIN` |

**`MATCH (p)-[i:投资]->(e)` 在单跳时就是一个 JOIN。** 图数据库的价值在**变长路径**，
八条里只有 C5 用到，而 C5 已用递归 CTE 实现（P=R=1.0，且与 NetworkX 交叉验证一致）。

→ 方案里「Cypher 表达力够写图约束」不构成上图数据库的理由 —— **反过来说 SQL 也够**。
→ 同理，`R-RY-01` 职业法人听起来像图算法，其实是 `COUNT(*) GROUP BY`，纯 SQL。

**图真正不可替代的只有三件事**（已全部实现于 `scripts/graph.py`）：
1. **穿透**（受益所有人）—— 多路径汇合要相加、环要剪枝、每跳要乘。递归 CTE 能做但会写成怪物
2. **社区发现**（Louvain）—— SQL 根本做不了
3. **子图提取** —— 证据链展示（前端用）

**原则 4「图谱先于标签」的表述也有问题**：我们没有任何图数据库就算出了股权环与同址聚集。
它真正想说的是「**关系必须先建模**」—— 而 `investment` / `position_hold` 本来就是边表，
关系早就建模了。「图谱」这个词混了三样东西：数据模型（PG 已是）/ 计算范式（需 NetworkX）/ 展示形式（前端）。

**选型结论：不上 Neo4j，用 NetworkX。**
数据量不需要（1000 家，内存里几十毫秒）；算法 NetworkX 全有且 Neo4j 社区版 GDS 是阉割的；
少一个数据库就少一套同步与一个 Demo 现场能挂的组件；NetworkX 无需部署，`pip install` 即可。

架构：**图是派生视图，不是第二个真相源** ——
`PostgreSQL（权威源）→ NetworkX（内存图）→ 写回 beneficial_owner + 标签库`，
「标签库是唯一出口」未被破坏。

### 3.9 ✅ 已核实：受益所有人穿透的正确论证（原「decl点 = 堵点」已证伪）
