# 数据接口规范（FastAPI）—— 看板 & Dify 智能助手的共用地基

> 实现：`scripts/api.py`（已实跑，6 个端点对真实数据验证通过）
> 启动：`.venv/bin/python scripts/api.py` → `http://127.0.0.1:8000`（交互文档 `/docs`）
> 数据源：PostgreSQL `reg`（标签库/指标库/画像视图），端点只读

---

## 0. 一条铁律（所有对接方必须遵守）

**LLM 不算数、不判定，只做路由 + 排版。** 判定来自规则引擎，数字来自本接口，
每个数字都带**口径 + 时点 + 版本**。三条硬约束写死在流程里：

1. 数字只能来自本接口返回，LLM 不许自己算、不许改写；
2. 每个数字答复必须带回它的口径（`口径`字段）和时点（`valid_from`）；
3. 报告里每条「因为…违反…」必须来自证据链字段（`证据` + `法条锚点`），不是生成。

## 1. 端点清单

| 方法 | 路径 | 用途 | 主要消费方 |
|---|---|---|---|
| GET | `/health` | 健康检查 | 运维 |
| GET | `/catalog/metrics` | **指标目录（语义层）** | Dify（映射问题→指标） |
| GET | `/catalog/tags` | 标签目录（含法条依据） | Dify / 报告 |
| GET | `/metric/{metric_id}?scope_value=` | **智能问数核心**：物化指标值+口径+时点 | Dify 问数 / 看板 |
| GET | `/enterprise/{ent_id}` | **企业风险画像一屏**（全息 JSON+证据链） | 看板证据链屏 / 报告典型案例 |
| GET | `/clusters?limit=` | 关联集群画像（按风险密度降序） | 看板中观 / 报告 |
| GET | `/clusters/{community_id}` | 单集群成员明细 | 看板下钻 |
| GET | `/report/situation` | **态势研判报告·结构化数据槽位** | Dify 报告 |

### 关键端点响应示例

**`GET /metric/M-LIVE-02`**（问数：直播电商许可缺失率）
```json
{
  "metric_id": "M-LIVE-02", "metric_name": "直播电商专题·许可缺失率",
  "口径": "直播专题企业中同时命中「许可项无批准文件」的占比……",
  "unit": "%", "grain": "中观", "source_type": "tag_agg",
  "data": [{ "scope_type": "topic", "scope_value": "直播电商",
             "value": 35.7143, "sample_size": 14,
             "valid_from": "2026-07-20T06:26:49Z", "metric_version": "v1" }],
  "_note": "数字为物化指标实例……LLM 仅排版，不得改写或再计算"
}
```

**`GET /enterprise/{ent_id}`**（微观画像，跨实体全息）
```json
{
  "企业": { "名称": "...", "状态": "存续", "注册资本": 300, "住所": "..." },
  "自身风险标签": [ { "标签": "许可项无批准文件", "类型": "法定强制",
                     "法条锚点": "《市场主体登记管理条例》第14条",
                     "证据": { "许可经营项目": ["食品销售"], "缺批准文件": true } } ],
  "关联风险": [ { "对象": "法定代表人", "标签": "失信人员违规任职",
                 "法条锚点": "《公司法》第178条第1款第5项……", "证据": {...} },
               { "对象": "住所", "标签": "住所批量聚集", "证据": {...} } ],
  "专题": ["直播电商专题"],
  "受益所有人穿透": [ { "自然人": "...", "持股比例": 1.0, "穿透路径": {...} } ]
}
```
> 「证据链即解释」：把 `法条锚点` + `证据` 倒着念就是「因某法条要求、某字段值为X，故命中」。LLM 只做通顺化。

**`GET /report/situation`**（报告：只出数据，不出散文）
```json
{ "宏观": { "月度新设趋势": [...] },
  "中观": { "直播电商专题看板": [...], "高风险关联集群": [...] },
  "微观": { "典型案例": [ <企业画像JSON>, ... ] },
  "生成约束": "每个数字/每条证据均来自物化资产；LLM 仅排版" }
```

## 2. 语义层清单（`/catalog/metrics` 的内容 —— 交给 Dify 做知识源）

Dify 的 LLM 读这张表，把自然语言问题映射到 `metric_id`，再调 `/metric/{id}`。

| metric_id | 名称 | 粒度 | scope_type | 单位 | 一句话口径 |
|---|---|---|---|---|---|
| M-NEW-01 | 月度新设经营主体数 | 宏观 | province | 户 | 按成立月计数（趋势） |
| M-CAP-01 | 新设主体注册资本中位数 | 宏观 | province | 万元 | 按月取中位数（长尾用中位数） |
| M-RISK-01 | 风险标签命中率 | 宏观 | province | % | 命中企业类风险标签的企业占比 |
| M-XN-01 | 登记机关退回率 | 中观 | authority | % | 已退回/(已核准+已退回) |
| M-XN-02 | 登记机关平均办理时长 | 中观 | authority | 日 | decided-submitted 均值 |
| M-ZS-01 | 聚集地址占比 | 中观 | province | % | 命中住所聚集的地址占比 |
| M-LIVE-01 | 直播电商专题·企业数 | 中观 | topic | 户 | 关键词圈定的直播企业数 |
| M-LIVE-02 | 直播电商专题·许可缺失率 | 中观 | topic | % | 直播企业中许可无批准文件占比 |
| M-LIVE-03 | 直播电商专题·合规问题企业占比 | 中观 | topic | % | 直播企业中命中任一法条风险占比 |
| M-CLU-01 | 关联集群·风险企业密度 | 中观 | community | % | 每个集群的风险企业占比（scope=集群号） |

> `scope_value` 取值：province→`32`；authority→`AU3201xx`；topic→`直播电商`；community→集群号（见 `/clusters`）。

## 3. Dify 对接：把接口配成「自定义工具」

Dify 工作流里，LLM 按问题类型路由，调不同工具：

| 用户问题类型 | LLM 调用 | 说明 |
|---|---|---|
| 法条含义 / 政策 | **法规 RAG**（组员的向量库） | 纯检索 |
| 指标/数据（「X 率多少」「几家」） | `/catalog/metrics` 定位 id → `/metric/{id}` | 数字来自库 |
| 某企业情况 | `/enterprise/{ent_id}` | 全息画像 |
| 某集群/团伙 | `/clusters` → `/clusters/{id}` | 中观下钻 |
| 出报告 | `/report/situation` → LLM 排版 | 数据槽位→散文 |

**给组员的 Dify 配置提示**：
- 把 `/catalog/metrics` 的返回塞进 Dify 知识库（或 system prompt），让 LLM 学会「问题→metric_id」的映射；
- 每个数据端点配成一个 HTTP 工具（Tool）；
- system prompt 里写死那三条铁律，尤其：**回答数字必须附口径和时点，禁止心算/估算**。

## 4. 前端对接（Next.js，给 C）

前端**一个页面两类内容**，只有对话框走 Dify：

- **三级分析看板**（趋势图/专题看板/集群图/企业画像/证据链/穿透图）→ **直连本接口**画图表，不经过 Dify：
  - 宏观趋势 → `/metric/M-NEW-01`、`/metric/M-CAP-01`
  - 直播专题看板 → `/metric/M-LIVE-01|02|03`
  - 集群图 → `/clusters`、`/clusters/{id}`
  - 企业画像/证据链屏 → `/enterprise/{ent_id}`
  - 股权穿透图 → `/enterprise/{ent_id}` 里的 `受益所有人穿透.穿透路径`（节点数组，喂 Cytoscape/G6）
- **智能助手对话框** → 嵌入 Dify（iframe 或 Dify chat API）

两者口径天然一致，因为 Dify 问数也是调同一套接口。

## 4b. 运行与网络（对接前必读，最常踩的坑）

**接口是常驻服务，必须运行中才能被调用**（进程一停，Dify/前端调用即失败）。一次完整 Demo 需同时开：
PostgreSQL(`pg-reg` 容器) + `api.py` + Dify(容器) + 前端(Next.js)。

**Dify（Docker）填工具 URL 的地址**（api.py 已绑 `0.0.0.0:8000`）：
- ✅ `http://host.docker.internal:8000`（Mac/Win Docker）或 `http://<主机局域网IP>:8000`（Linux）
- ❌ 不要用 `localhost` / `127.0.0.1` —— 在容器里指容器自己，连不上
- 浏览器端前端在主机跑，可用 `localhost:8000`（CORS 已放开）

**若用 Dify 云版**（非自托管）：云端够不到你本机 `localhost`，需 `ngrok http 8000` 内网穿透或部署到公网。

**保持运行**：Demo 期 `.venv/bin/python scripts/api.py`（别关窗口）；后台 `nohup … &`；正式上 systemd/compose。
