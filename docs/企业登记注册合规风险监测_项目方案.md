# 基于数据标签挖掘的企业登记注册合规风险监测

## 项目方案

---

## 0. 摘要

本方案面向"总局统筹、属地办理"体制下企业登记注册的合规保障需求，构建一套**以证据链为核心、以规则库为中枢、以知识图谱为骨架**的合规风险监测系统。

**一句话定位**：把总局当前"定期人工出具的数据质量情况报告和数据分析报告"，变成一个**实时、可追溯、可自我更新**的监管规则中台。

**与常规方案的根本区别**：本方案不是一个"更准的风险评分模型"，而是一个"**更可信的判定机制**"。在监管场景中，precision 从 0.91 提升到 0.94 无人在意；"你凭什么这么判"和"判错了怎么办"才是全部。因此系统的全部设计围绕三条约束展开：

1. **判定必须确定性、可复现、可追责** —— 大模型不参与任何判定，只做旁路的解释、问答与报告；
2. **每一条结论必须能回溯到法条** —— 无法源即无判定；
3. **规则是数据不是代码** —— 规则变更不发版，误报自动沉淀为规则更新建议。

## 1. 赛题理解

### 1.1 题面拆解

赛题背景中隐含了四个并列的病灶，它们不是同一层问题：

| 赛题原文 | 问题层级 | 本方案对应机制 |
|---|---|---|
| 各地校验标准不一 | **机制问题** | 统一规则中台 + 属地差异化参数 |
| 分散核验缺乏全国统一管控 | **数据问题** | 全国图谱 + 跨省图约束一致性检测 |
| 虚假申报、数据矛盾 | **识别问题** | 规则引擎 + 图算法 + 形式化约束冲突检测 |
| 规则更新滞后 | **治理问题** | 规则即数据 + 误报回流生成目录更新建议 |
| 系统承载不足 | **工程问题** | 触发时点即算力分层：事前 O(在途量) / 事后 O(存量)，办件延迟不随存量增长（见 §6.4，含 EXPLAIN 实测） |

**关键判断**：多数参赛方案会把全部精力投向第三行（识别问题），因为那是可以卷模型的地方。但赛题背景里第一、第四行才是反复强调的痛点，且**没有任何一条能靠模型解决**。本方案将机制设计置于模型精度之上。

### 1.2 三个项目目标的技术翻译

| 赛题目标 | 技术含义 |
|---|---|
| 目标1：梳理法规与两级流程，挖掘堵点、风险与效能短板 | 法规 → 七列规则表；流程 → 数据血缘；**"填不出判定逻辑的规则行"即堵点** |
| 目标2：数据治理 + 标签与分析预警体系，宏观-中观-微观立体研判 | 微观看**标签**、中观看**指标**、宏观看**趋势**——三者是同一张标签实例表的不同聚合粒度 |
| 目标3：智能问答、智能问数、智能报告 | 法规 RAG（带条款引用）+ 图谱 KBQA + 证据驱动的报告生成 |

## 2. 业务背景梳理（对应目标 1）

### 2.1 制度框架

企业登记注册实行 **"总局统筹、属地办理"** 双层体制：

- **属地办理**：省 / 市 / 县三级登记机关通过各自的全程电子化登记系统受理与核准。全国各省级系统入口汇集于**全国经营主体登记注册服务网**（dj.samr.gov.cn）。
- **总局统筹**：总局制定统一的登记业务规范、数据标准与平台服务接口；建立**全国统一的企业登记数据库**，归集全国经营主体登记管理信息，并通过**国家企业信用信息公示系统**（gsxt.gov.cn）对外公示。

> **实证素材（PPT 第一页用）**：打开 dj.samr.gov.cn，随机进入 5 个省的登记入口，截图对比申报表单字段差异。这就是"各地校验标准不一"最直观的证据——不是推论，是照片。

### 2.2 正向办理流程（一笔登记业务）

```
申请人 / 登记代理人
   │  实名认证、代理身份备案
   ▼
地方登记机关受理（省 / 市 / 县属地办理）
   │
   ▼
形式审查与规则校验  ◄────►  【总局统一校验接口】
   │                          · 实名 / 代理人核验
   │                          · 名称禁限用规则
   │                          · 经营范围规范表述
   │                          · 外资准入负面清单
   ▼
核准登记、发放执照（电子营业执照即时生成）
   │
   ▼
数据实时汇集总局（"一登记、即公示"）
```

**关键机制**：校验能力是**通过总局开发的统一应用接口下发**的，而非各地自建。例如：

- 各级登记机关办理外资登记时**必须调用**总局的外资登记负面清单禁限要求校验接口，根据接口反馈做禁限制登记；
- 经营范围登记**必须调用**总局统一接口，使用《经营范围规范表述目录》；地方发现新兴行业经营项目后**向总局反馈**，由**总局定期更新维护**规范目录。

> **"规则更新滞后"的真正含义就在这里**：规则挂在总局接口上，目录靠总局定期维护。新业态出现时，地方只能"反馈"然后等待。这是一个**规则治理机制问题，不是模型问题**。本方案的第一创新点正面回答这一条。

### 2.3 整改回传闭环（反向流程）

```
全国统一登记数据库（归集登记数据 + 过程数据）
   │
   ▼
数据质量监测分析（依据《市场主体登记数据质量建设实施方案》及质量评价标准）
   │
   ▼
生成疑点问题清单 ──────►  【典型疑点】
   │  按省下发                · 冒名 / 身份异常
   │                          · 住所批量聚集
   │                          · 出资与注册资本不符
   │                          · 名称含禁限用词
   ▼
地方核实与整改（更正、补录、撤销登记）
   │
   ▼
整改结果回传总局（复核销号）
   │
   └──► ↻ 销号后数据回流数据库，形成闭环
```

**政策依据**：省级市场监管部门应当执行总局制定的《市场主体登记数据质量建设实施方案》及相关数据质量评价标准，健全本地区数据质量监测机制；**总局对汇集数据进行监测分析，定期发布数据质量情况报告和数据分析报告**。

> **这句话定义了本项目的坐标**：赛题目标 3 的"智能报告"，对应的正是这份总局本来就在人工出具的报告。**本项目不是发明新东西，是把"定期人工出报告"变成"实时自动出报告，且每个结论可点开溯源"**。这个叙事在答辩中极具说服力，因为它证明方案对接的是真实存在的业务动作，而非想象出来的需求。

### 2.5 监管堵点、风险与效能短板（目标 1 的产出）

#### 官方点名的虚假登记类型（直接采用，不自行编造）

- **被法人**（身份被冒用登记为法定代表人）
- **假国企**（虚构国资背景股权链）
- **假事业单位**
- **国字头协会学会**

#### 规则滞后的活案例

2026 年新规明确要求：企业、个体工商户注册名称和经营范围中不得含有"虚拟货币""虚拟资产""加密货币""加密资产""稳定币""现实世界资产代币化"等表述。

> **从新业态出现，到禁用词进入总局规则库，中间的时间差，就是本系统要压缩的东西。**这是最好的 Demo 素材：评委一看就懂。

#### 效能短板（最易被忽略）

赛题白纸黑字写了"效能短板"，但**效能标签不挂在企业身上**，它挂在**业务单据**和**登记机关**上：

- 退回率、退回原因分布
- 整改闭环时长分布
- 规则误报率（按省、按规则）
- 事前拦截 vs 事后回溯的比例

> **想通这一点，即可领先大部分队伍**——所有人都在给企业打标签，没人给监管流程本身打标签。

## 3. 总体方案设计

### 3.1 设计原则

| 原则 | 含义 | 反面案例 |
|---|---|---|
| **证据先于结论** | 判定由规则命中产生，证据链即解释本身 | 先出风险分数，再让 LLM 编一个解释（post-hoc rationalization，监管场景致命） |
| **LLM 严格旁路** | 大模型不产生任何新判断，只做翻译、检索、排版 | 把 LLM Agent 放在"风险检测 → 风险处置"之间的决策主链上 |
| **规则即数据** | 规则存库，引擎只做解释器；改规则不发版 | 规则硬编码在代码里，法规一变就要重构 |
| **图谱先于标签** | 关系类标签必须先有图谱才算得出来 | 把"标签生成"排在"知识图谱"之前——这会砍掉本赛题一半的价值 |
| **只读标签库** | 下游应用只读标签库，不直接读原始数据 | 驾驶舱说 2.3%、专题页说 3.1%，口径打架 |
| **监测已发生 ≠ 预测未来** | 识别已发生的行为序列，不预测谁"将要"违法 | "提前预警"——人家还没做，你凭什么预警？法律依据在哪？ |

### 3.2 系统主干管线

```
┌──────────────┬──────────────┬──────────────┐
│ 登记业务数据  │ 法规政策文本  │ 外部关联数据  │   ← 输入层
│ 申报/核准/退回│条例/公司法/目录│许可/失信/行业码│
└──────┬───────┴──────┬───────┴──────┬───────┘
       └──────────────┼──────────────┘
                      ▼
        ┌─────────────────────────────┐
        │        数据归集治理          │   ← 治理层
        │ 标准化·清洗·实体对齐·主数据  │
        └──────────────┬──────────────┘
                       ▼
  ┌────────────────── 知识层 ──────────────────┐
  │ 业务知识图谱   │  法规向量库  │   规则库   │
  │企业·人·地址·代理│ 条文切片可引用│ 七列规则表 │
  └────────────────────┬───────────────────────┘
                       ▼
  ┌────────────────── 计算层 ──────────────────┐
  │  规则引擎     │    图算法    │  冲突检测   │
  │  确定性校验   │ 聚集度/股权环 │ 图约束违反  │
  └────────────────────┬───────────────────────┘
                       ▼
        ┌─────────────────────────────┐
        │           标签库             │   ← 全系统的"腰"
        │ 实体 + 标签 + 时点 + 证据ID  │      唯一数据出口
        └──────────────┬──────────────┘
              ┌────────┴────────┐
              ▼                 ▼
     ┌─────────────┐    ┌─────────────┐
     │ 三级分析引擎 │    │  智能助手   │   ← 应用层
     │趋势·指标·预警│    │问答·问数·报告│
     └──────┬──────┘    └──────┬──────┘
        ┌───┴───┐              │
        ▼       ▼              ▼
   ┌────────┐┌────────┐  ┌──────────┐
   │风险预警││质量问题│  │态势研判  │   ← 输出层
   │  工单  ││  清单  │  │  报告    │
   │阻断/复核││按省下发│  │结论可溯源│
   │ /打标  ││  整改  │  └──────────┘
   └────┬───┘└───┬────┘
        └────┬───┘
             ▼
   ┌──────────────────────┐
   │   人工复核结果回流    │
   │  误报标注 → 规则调优  │
   └──────────┬───────────┘
              └──► ↻ 回流规则库，形成自优化闭环
```

**三个必须理解的设计决定**：

1. **标签库是全系统的"腰"**。上游一切算法的唯一产物是标签实例；下游一切应用只读标签库。这条约束保证三级分析用同一套口径。

2. **证据 ID 是命脉**。每条标签实例必须挂一个 evidence_id，指向四样东西：命中的**规则版本号** + 涉及**字段值快照** + **图谱子图** + **法条锚点**。规则版本号尤其关键——它让系统能回答"半年前这条为什么没报？"（因为当时规则是 v1）。

3. **最后那个回流箭头是方案的灵魂**。没有它，这只是一个报警器；有了它，这是一个会自我进化的规则治理系统。

### 3.3 与通用数据中台方案的区别

一个典型的错误方案长这样：

```
企业登记申请 → 数据接入 → 数据治理 → 标签生成 → 知识图谱
            → 风险检测 → LLM Agent 分析 → 风险处置
```

它的四个问题，每一个都会在答辩现场被打穿：

| 问题 | 后果 |
|---|---|
| **标签生成排在知识图谱之前** | 关系类标签（同址聚集、股权环、跨省任职）必须先有图谱才算得出。顺序反了，直接损失一半含金量 |
| **只有"企业登记申请"一个输入，无法规、无规则库** | 那"合规"从哪来？没有法条锚点，风险检测凭什么判？做出来的是"异常检测"，不是"合规风险监测" |
| **LLM Agent 站在决策主链上** | 评委问"大模型误判导致企业开不了业，谁担责"——无解 |
| **单向到底，无闭环、无整改回传** | 缺了赛题原文点名的两级流程组成部分 |

其本质是**一个通用数据中台模板换了领域词**——做电商风控、做反欺诈，同一张图。它没有回答本赛题的任何一个特有问题。

## 4. 数据归集治理

### 4.1 数据来源策略（出题方不提供数据）

三条腿走路，按可信度排序：

| 来源 | 内容 | 规模 | 用途 |
|---|---|---|---|
| **真实公开数据** | 国家企业信用信息公示系统（gsxt.gov.cn）：企业名称、统一社会信用代码、董事、股东、成立日期、注册地址、经营范围 | **手工采集 200–500 家** | 验证 schema、Demo 真实性展示 |
| **合成数据（主力）** | 按规则表反向注入异常：同址批量、股权环、出资不符、禁用词、跨省任职 | 10 万级 | **唯一有 ground truth 的数据**，用于评测 |
| **政策语料（100% 真实）** | dj.samr.gov.cn 政策专栏 + 各法规原文 | 数十条问答 + 十余部法规 | 智能问答 RAG 语料 |

> **对评委的话术**：**"数据是合成的，但规则、流程、法条引用全部真实可溯源；且正因为是合成的，我们才有 ground truth，才能报出准确率和召回率——这是使用真实数据的队伍做不到的。"**
>
> 把限制讲成优势，且这个优势是真的。绝大多数队伍拿不到真实标注，只能定性描述效果。

### 4.2 合成数据生成方法

1. **骨架真实**：以公开采集的 200–500 家真实企业的字段分布（行业分布、注册资本分位数、地域分布、经营范围词频）为参数，拟合生成器；
2. **异常注入**：按七列规则表逐条构造正样本，每条规则至少 50 个正例 + 对应的**难负例**（接近阈值但合规）；
3. **标注即生成**：注入时记录 (entity_id, 违反的规则 id, 注入参数) → 即为 ground truth；
4. **污染控制**：设置 3%–5% 的**标注噪声**（模拟真实场景中复核员也会判错），使评测不至于虚高。

> 第 4 条是加分项：主动引入噪声、主动报告噪声下的性能衰减，展现学术训练痕迹。

### 4.3 实体对齐（治理层的真正难点）

这是被通用方案完全隐藏、但决定图谱质量的一步：

| 实体 | 对齐依据 | 难点 |
|---|---|---|
| 企业 | 统一社会信用代码 | 相对容易；注意历史名称、注销后重名 |
| 自然人 | 证件号哈希 | 同名不同人、跨省证件类型不一 |
| **地址** | 标准化到**门牌粒度** | **最难**。"XX路88号2001室" / "XX路88号20层01" / "XX路88弄2001" 需归一到同一 address_id。这一步做不好，"同址聚集"全部失效 |
| 代理人 | 全国代理人库 ID | 2025 新规后有全国统一 ID，相对可靠 |
| 经营范围条目 | 规范表述目录 ID | **对不上的即为目录缺口**（见 §9.4） |

> **地址标准化是本项目的隐形胜负手**。建议投入的工作量不低于图谱建模本身。可采用"规则归一 + 地址要素切分 + 编辑距离聚类"三步法，并在文档中报告归一准确率。

## 5. 知识层

### 5.1 业务知识图谱 Schema

**实体（6 类）**

| 实体 | 关键属性 |
|---|---|
| 企业 / 经营主体 | 统一社会信用代码、类型、注册资本、成立日期、状态、登记机关 |
| 自然人 | 证件号哈希、姓名哈希 |

| 登记代理人 / 代理机构 | 全国代理人 ID、所属机构、备案状态 |

> 另有两类"非典型"实体，是效能分析的挂载对象：**业务单据**（申请件）与**登记机关**。

**关系**

| 关系 | 起点 → 终点 | 用途 |
|---|---|---|
| 投资 / 出资 | 自然人/企业 → 企业 | 股权穿透、股权环 |
| 任职 | 自然人 → 企业 | 职业法人、跨省任职 |
| 注册于 | 企业 → 地址 | 同址聚集 |
| 代办 | 代理人 → 业务单据 | 代办黑产识别 |
| 经营 | 企业 → 经营范围条目 | 超范围、目录缺口 |
| 持有 | 企业 → 许可证 | 许可缺失检测 |
| 变更为 | 企业 → 企业(历史版本) | 行为序列 |
| 受益所有 | 自然人 → 企业 | 穿透识别 |

### 5.2 法规向量库

- **切片粒度：条 / 款级**，不是段落级。因为引用必须精确到"《条例》第 15 条"，段落级切片会导致引用漂移。
- **每个切片挂 `law_anchor`**：`{法规名, 版本, 条, 款, 生效日期, 失效日期}`。生效/失效日期字段是关键——它让系统能回答"该企业设立于 2023 年，当时适用的是修订前的规则"。
- **检索策略**：向量召回 + 法规名/条号的字面精确匹配混合。纯向量在法条场景召回不稳。

### 5.3 规则库：七列规则表

这是全项目**最硬的资产，比代码值钱**——它是唯一一个别人抄不走、且能证明团队真读了法规的东西。

| 列 | 说明 | 为什么必须有 |
|---|---|---|
| **法条锚点** | 《条例》第 15 条 | **堵住幻觉**。没有这列，规则就是拍脑袋的 |
| **字段** | 涉及的数据字段 | 去重后即最小数据模型 |
| **判定逻辑** | 可执行表达式 | **填不出来的行 = 监管堵点**（见下） |
| **风险标签** | 命中后产生的标签 | 去重后即标签体系初稿 |
| **规则类型** | 法定强制 / 规范性 / 统计异常 / 情报线索 | "同址聚集 500 家"**没有直接法条**，硬安一个会被业务评委当场戳穿 |
| **触发时点** | 事前拦截 / 事中 / 事后回溯 | 同一规则在不同时点是完全不同的产品 |
| **处置级别** | 硬阻断 / 人工复核 / 仅打标 | 决定误报代价。全设"阻断"是灾难 |

**这条链正着读是设计，倒着读是解释。**系统报警时复核人员只问一句"凭什么"，答案必须是"因为《XX条例》第X条要求Y，该企业字段Z违反了，故命中标签W"——这句话就是四列倒着念。

**四个真实收益**：

1. **堵住幻觉**：每条规则必须报出法源；
2. **规则更新变成机械操作**：法条变了 → grep 法条列 → 找到受影响规则 → 改。这就是"规则更新滞后"的工程回答；
3. **空白格就是发现**：填不出"判定逻辑"的行，要么数据拿不到（如住所真实性需实地核查），要么它根本不是校验而是裁量。**这些空白格就是目标 1 要的"监管堵点"**——不是失败，是结论；
4. **顺手生成两个交付物**：字段列去重 = 数据模型；风险标签列去重 = 标签体系初稿。

**规则表样例（节选）**

| 法条锚点 | 字段 | 判定逻辑 | 风险标签 | 规则类型 | 触发时点 | 处置级别 |
|---|---|---|---|---|---|---|
| 《公司法》第47条 | 股东认缴出资额、注册资本 | `SUM(股东认缴) != 注册资本` | 出资与注册资本不符 | 法定强制 | 事前拦截 | 硬阻断 |
| 《公司法》第47条 | 出资缴付期限、成立日期 | `缴付期限 > 成立日期 + 5年` | 出资期限超限 | 法定强制 | 事前拦截 | 硬阻断 |
| 《条例》第15条 | 法定代表人证件号 | 实名核验接口返回失败 | 实名核验不通过 | 法定强制 | 事前拦截 | 硬阻断 |
| 《条例》第14条 | 经营范围条目、许可证 | `经营范围含许可项 AND NOT EXISTS(许可证节点)` | 许可缺失 | 法定强制 | 事中 | 人工复核 |

| 《经营范围规范表述目录》 | 经营范围条目 | `条目 NOT IN 规范目录` | 表述不规范 | 规范性 | 事中 | 仅打标 |
| — （虚假登记推定） | 地址 ID、企业数 | `同址主体数 > P99 AND 30日新增 > 阈值` | 住所批量聚集 | **统计异常** | 事后回溯 | 人工复核 |
| — （虚假登记推定） | 代理人 ID、代办量 | `代办量 > 3σ AND 核准率 < 50%` | 代办行为异常 | **统计异常** | 事后回溯 | 人工复核 |

## 6. 计算层

三条并行路径，分别对应三类规则：

### 6.1 规则引擎（确定性校验）

- **规则存库，引擎只做解释器**。判定逻辑以表达式 DSL 或 SQL 片段存储于 `RULE` 表，引擎解析执行。
- **改规则不发版**——这是"规则即数据"落地的技术形态，也是对赛题"规则更新滞后"的正面回应。
- 每次执行写入 `rule_version`，保证历史可复现。

### 6.2 图算法（统计异常）

| 算法 | 检测目标 |
|---|---|
| 同址聚集度（度中心性 + 时间窗增量） | 住所批量注册 |
| 社区发现（Louvain） | 关联企业集群 |
| 强连通分量 | 股权环 |
| 跨省任职计数 | 职业法人 |
| 代办量时序突变（3σ / 变点检测） | 代办黑产 |

**阈值必须来自数据分位数（如 P99），不能拍脑袋**，且需报告稳定性（见 §7.5）。

### 6.3 图约束冲突检测（本方案核心技术创新）

**关键认知**：赛题原文写的是"**数据矛盾**"，不是"风险传播"。

**数据矛盾是约束违反，不是概率问题。**它不需要拟合、不需要打分，用形式化图约束（SHACL / Datalog / Cypher 约束）即可检测，**零误报、100% 可解释**——因为它不是打分，是逻辑冲突。

而"属地办理"必然导致跨省登记数据互相矛盾——**这正是赛题背景那句"分散核验缺乏全国统一管控"的直接后果**。

**八条图约束（形式化 + 法源 + 检测逻辑）**

| # | 约束（自然语言） | 法源 | 检测逻辑（Cypher 风格） |
|---|---|---|---|
| C1 | 股东认缴出资之和必须等于注册资本 | 《公司法》第47条 | `MATCH (p)-[i:投资]->(e) WITH e, SUM(i.认缴额) AS s WHERE s <> e.注册资本 RETURN e` |
| C2 | 失信被执行人不得在任何省新任法定代表人 | 《公司法》任职资格 + 失信名单 | `MATCH (p:自然人 {失信:true})-[t:任职 {职务:'法定代表人'}]->(e) WHERE t.生效日 > p.失信起始日 RETURN p, e` |
| C3 | 经营范围含许可项则必须存在有效许可证节点 | 《条例》第14条 | `MATCH (e)-[:经营]->(s:条目 {许可项:true}) WHERE NOT (e)-[:持有]->(:许可证 {类型:s.许可类型, 有效:true}) RETURN e` |
| C4 | 已注销企业不得有新增变更登记 | 《公司法》第37条 | `MATCH (e {状态:'注销'})-[v:变更为]->(e2) WHERE v.日期 > e.注销日 RETURN e` |
| C5 | 股权关系不得成环 | 《公司法》/《受益所有人办法》 | `MATCH path=(e)-[:投资*1..6]->(e) RETURN path` |
| C6 | 同一自然人不得同时是登记代理人与登记联络员（自行出资设立的除外） | 《代理行为管理办法》 | `MATCH (p)-[:代办]->(d)-[:关于]->(e), (p)-[:联络员]->(e) WHERE NOT (p)-[:投资]->(e) RETURN p` |
| C7 | 特种设备生产类经营范围必须匹配特种设备生产许可 | TSG 07—2019 | `MATCH (e)-[:经营]->(s:条目) WHERE s.行业码 IN 特种设备生产码表 AND NOT (e)-[:持有]->(:许可证 {类型:'特种设备生产'}) RETURN e` |
| C8 | 同一统一社会信用代码不得对应多个不同名称的有效主体 | 代码唯一性 | `MATCH (e1), (e2) WHERE e1.代码 = e2.代码 AND e1.名称 <> e2.名称 AND e1.有效 AND e2.有效 RETURN e1, e2` |

> **为什么别人做不了**：因为他们没意识到"数据矛盾"是个**图约束问题**，会当成异常检测去拟合。一旦拟合，就有了误报，就要解释，就陷入不可解释的泥潭。
>
> **C2 和 C8 是跨省场景的杀手锏**——单省数据里根本看不出矛盾，只有全国图谱才暴露。这直接坐实了"分散核验缺乏全国统一管控"的危害，并证明"统一管控"的价值。

> **★ 实测修正（2026-07-17）**：把这八条的 Cypher 逐条拆开后发现，**只有 C5（股权环）真正用到变长路径** `-[:投资*1..6]->`；其余七条单跳即 JOIN——C1 是 `GROUP BY HAVING`，C8 甚至连一条边都没有（纯自连接）。`MATCH (p)-[i:投资]->(e)` 在单跳时就是一个 JOIN。
>
> 因此"Cypher 表达力够写图约束"**不构成上图数据库的理由**——反过来说 SQL 也够。C5 已用 PostgreSQL 递归 CTE 实现（P=R=1.0，且与 NetworkX 的强连通分量交叉验证结果一致：两条独立实现均得 14 家环上企业）。
>
> **图真正不可替代的只有三件事**：①穿透（受益所有人，多路径汇合相加 + 环剪枝 + 每跳累乘）；②社区发现（Louvain，SQL 做不了）；③子图提取（证据链展示）。选型据此定为 **NetworkX 而非 Neo4j**：1000 家节点内存里几十毫秒跑完，无需部署，Demo 少一个能挂的组件；且图是**派生视图不是第二真相源**（`investment` 表本来就是边表）。

### 6.4 承载分流：触发时点即算力分层（对应赛题"系统承载不足"）

赛题背景点名了五个病灶，"系统承载不足"是最容易被当成运维问题忽略的一个。**但它不需要新机制——七列规则表里的"触发时点"列本身就是解法**，只是过去没被这样讲。

**机理**：两类规则的复杂度曲线根本不同。

| | 事前拦截（`R-CZ-01-PRE`） | 事后回溯（`R-CZ-01`） |
|---|---|---|
| 判定对象 | 申请件 `application`（在途） | 已登记企业 `enterprise`（存量） |
| 扫描方式（实测 EXPLAIN） | **Index Scan** | **Seq Scan** × 2 |
| 实际扫描 | 50 条在途申请 | 1000 企业 + 1907 出资 |
| Buffers | **3** | **47** |
| **复杂度** | **O(在途量)** | **O(存量)** |

**关键不是 15.7 倍这个数字，是两条曲线的斜率不同。** 在途量是稳态的（每天新申报量大体恒定），而存量只增不减。

> **若把所有规则都放在事前，每笔办件都要触发一次 O(存量) 全表扫描 —— 办件延迟将随存量线性增长，系统越用越慢，最终卡死。这就是"系统承载不足"的机理。**
>
> 分层之后，**办件延迟只与在途量相关，与存量无关**。这不是靠堆机器，是靠让 99% 的办件只经过轻量路径。

**一条必须守住的约束**：**规则声明的触发时点，必须与其判定逻辑实际作用的对象一致。**

- 事前拦截 → 只能查 `application`（此时统一社会信用代码尚未发放，企业实体**不存在**）
- 事后回溯 → 查 `enterprise`

违反这条会同时产生两个后果：**承载分层失效**（事前规则退化成全表扫描），以及**处置动作无意义**（企业都登记完了，"硬阻断"阻断谁？）。原型期即踩过此坑——`R-CZ-01` 曾照本节上文的规则表样例标为"事前拦截 + 硬阻断"，但其 logic 查的是 `enterprise`，实为事后清存量；已修正为"事后回溯 + 人工复核"。

> **事前防新增，事后清存量。**同一条法条（《公司法》第47条）、同一个风险标签，在两个时点是两条规则、两种处置、两种复杂度 —— 这正是 §5.3 那句"**同一规则在不同时点是完全不同的产品**"的落地形态。

## 7. 标签体系（对应目标 2）

### 7.1 核心认知：标签体系是结果，不是前提

不要一上来画一棵漂亮的标签树，那必然是空中楼阁。它由两股力量夹逼出来：

- **自下而上**：规则表的每一行命中 → 一个候选标签
- **自上而下**：宏观/中观/微观三级分别要回答什么问题 → 反推需要什么维度

两边对撞、去重、补缺，收敛出来的才是真的。**这个"对撞"过程本身就是方法论贡献，写进文档即为亮点。**

### 7.2 标签 / 特征 / 指标之辨

| 概念 | 定义 | 例子 |
|---|---|---|
| **标签 (tag)** | 挂在**实体**上的离散判断，人能读懂 | 该企业命中 · 住所批量聚集 |
| **特征 (feature)** | 喂给模型的数值 | 同址企业数 = 487 |
| **指标 (metric)** | 挂在**群体**上的聚合统计 | 全省同址聚集企业占比 2.3% |

> **"宏观-中观-微观"的技术含义就是这个映射：微观看标签，中观看指标，宏观看趋势。**这一句能让评委确信团队真的想清楚了。

### 7.3 挂载对象（最容易漏、最能拉开差距的一步）

| 挂载对象 | 标签举例 | 差异化程度 |
|---|---|---|
| 企业 | 出资不符、许可缺失、名称禁用词 | 大家都会做 |
| 自然人 | 跨省任职数、被冒名嫌疑、职业法人 | 官方点名的"**被法人**"挂这里 |
| **登记代理人** | 批量代办、异常核准率、跨省代办 | **2025 新规刚把它变成监管抓手，几乎没人会做** |
| 地址 | 聚集度、虚拟地址、增量突变 | 图谱的关键节点 |
| **业务单据 / 流程** | 退回次数、整改时长、拦截时点 | **"效能短板"挂这里，不挂企业** |
| **登记机关** | 数据质量得分、规则误报率、退回率排名 | "各地标准不一"的量化载体 |

> 后三行是本方案的差异化。**赛题白纸黑字写了"效能短板"，但效能标签根本不挂在企业身上。**

### 7.4 标签库数据模型（四张表）

标签库不是一张表，是四张表。分不清这四张，就一定会做成一坨。

```
RULE  ──产出──►  TAG_DICT  ──定义──►  TAG_INSTANCE  ◄──挂载──  ENTITY
                                            │
                                          支撑
                                            ▼
                                        EVIDENCE
```

**`RULE`（规则库）**
| 字段 | 说明 |
|---|---|
| rule_id (PK) | |
| law_anchor | 法条锚点 |
| rule_type | 法定强制 / 规范性 / 统计异常 / 情报线索 |
| trigger_point | 事前 / 事中 / 事后 |
| disposal_level | 阻断 / 复核 / 打标 |
| logic | 可执行表达式 |
| rule_version | |

**`TAG_DICT`（标签字典 · 定义层 · 静态）**
| 字段 | 说明 |
|---|---|
| tag_id (PK) | |
| tag_name | |
| entity_type | 挂载对象 |
| value_type | 布尔 / 枚举 / 数值 |
| action | **命中后谁做什么**（填不出就砍掉这个标签） |
| basis_type / basis_ref | 依据类型（法条 / 统计基线）与引用 |
| update_freq | 更新频率 |
| expiry_cond | 失效条件 |
| tag_version | |

**`TAG_INSTANCE`（标签实例 · 事实层 · 动态）**
| 字段 | 说明 |
|---|---|
| instance_id (PK) | |
| tag_id (FK) / entity_id (FK) | |
| value | |
| **valid_from / valid_to** | **时点，绝不覆盖写（SCD Type 2）** |
| confidence | |
| evidence_id (FK) | |

**`EVIDENCE`（证据）**
| 字段 | 说明 |
|---|---|
| evidence_id (PK) | |
| rule_version | 当时的规则版本 |
| field_snapshot | 字段值快照 (JSON) |
| subgraph_ref | 图谱子图引用 |
| law_anchor | 法条引用 |

**三个 schema 决定是关键**：

1. **`TAG_INSTANCE` 必须有 `valid_from/valid_to`，绝不覆盖写。**因为监管必须能回答"去年这家企业当时是什么状态"。而且——**宏观趋势不需要另建统计表，趋势就是这张表按时间聚合**。一个设计解决两个需求。
2. **`tag_version` 与 `rule_version` 必须分开。**标签定义变了（"住所聚集"口径改了）vs 规则实现变了（阈值从 500 调到 200），是两回事，追责时会问到。
3. **`confidence` 不是模型分数**，是规则类型决定的先验。**法定强制规则 `confidence = 1.0`，没有商量余地。**把它当模型输出，就把确定性规则污染成概率了。

### 7.5 构建顺序（五步，别跳步）

1. 从七列规则表的"风险标签"列**去重** → 候选标签清单（自下而上）
2. 从三级分析需求**反推**缺什么维度 → 补（自上而下）
3. 把标签**分配到 6 类挂载对象**上 —— 这一步会暴露"只做了企业标签"
4. 每个标签填**元数据卡**，**卡片填不满的当场砍掉** —— **这一步是筛子，80% 的垃圾标签死在这里**
5. 写算子、跑通、产出实例

**数量控制：60–80 个精标签，其中 20 个核心风险标签有完整评测。**不做 500 个。评委看深度不看数量；标签一多，第一个被问的就是"你怎么验证的"。

### 7.6 合理性检验（三层）

**不用"看起来挺全"这种验收标准。**

#### 第一层：结构性检验（无需数据）

| 检验 | 做法 | 不合格信号 |
|---|---|---|
| 需求覆盖 | 做「赛题要求 × 标签」矩阵 | 有空行 = 漏了赛题点名的东西 |
| 法源覆盖 | 第一梯队法规每个"应当/不得"是否被覆盖 | 未覆盖的须说清是"漏了"还是"不可计算" |
| 挂载完整 | 6 类实体是否都有标签 | **只有企业标签 = 直接不合格** |
| 正交性 | 标签两两语义重叠？画共现矩阵 | 高共现的合并，或说明为何保留 |

#### 第二层：数据性检验（有合成数据即可做）

| 检验 | 健康区间 | 不合格信号 |
|---|---|---|
| **命中率分布** | 风险标签 **0.1%–5%** | 命中 0 家 = 死标签（规则写错/阈值离谱）；命中 90% = 无区分度 |
| 稳定性 | 换月数据波动 < 50% | 波动大 = 阈值是拍的不是算的 |
| 准召率 | 对 20 个核心标签报 P/R/误报率 | 用合成数据 ground truth |
| 噪声鲁棒性 | 3%–5% 标注噪声下性能衰减可接受 | 衰减剧烈 = 过拟合 |

> **命中率分布这条最实用，一跑就知道哪些标签是垃圾。**

#### 第三层：业务性检验（最重要，也最没人做）

| 检验 | 做法 |
|---|---|
| **可行动检验** | 每个标签问"命中了谁做什么"。答不上 → 砍 |
| **告警负荷检验** | **最容易翻车的一条，务必做**：全部标签跑一遍，一天产生多少条工单？除以复核人力。80 个标签一天报 5 万条，一个省登记科 20 人，人均 2500 条/天 —— **系统当场废掉**。把这个数字算出来放进文档，评委立刻知道你懂业务 |
| **误报代价分层** | 阻断级标签必须 **precision > 0.99** 才配设为阻断，否则降级为复核。**这条写进方案，就把"误报怎么办"这个必问题提前答了** |

#### 30 秒最小合理性测试

每个标签必须能一句话答出三个问题：

1. 它挂在**谁**身上？
2. 它的**依据**是什么（法条 or 统计基线）？
3. 命中后**谁做什么**？

**答不齐这三个，那不是标签，是幻觉。**用这三问扫一遍清单，能砍掉一半。

## 8. 宏观-中观-微观三级研判体系

三级不是三套系统，是**同一张 `TAG_INSTANCE` 表的三种聚合粒度**。

### 8.1 宏观层：（法人）态势研判

| 视图 | 内容 |
|---|---|
| 新设 / 注销趋势 | 按月、按省、按行业门类 |
| 资本结构演变 | 注册资本分位数迁移 |
| 行业迁移 | 经营范围条目词频变化（新业态雷达） |
| **数据质量总分** | 按省的质量评价得分排名 |
| **风险态势指数** | 各类风险标签命中率的时序 |

### 8.2 中观层：重点领域分析

依据《市场监管总局关于加强重点领域信用监管的实施意见》中的**重点领域企业类型清单**划定范围。

| 专题 | 指标 |
|---|---|
| 异常地址集群 | 聚集度 Top N、增量突变地址 |
| 异常代办机构 | 代办量 Top N、低核准率机构 |
| 假国企 / 国字头 | 名称含国资特征词但股权链穿透不到国资 |
| 跨省关联集群 | 社区发现结果中跨 ≥3 省的集群 |

### 8.3 微观层：专题预警（赛题点名的两个）

#### 专题一：特种设备

| 检测点 | 依据 |
|---|---|
| 经营范围含特种设备生产项 但无特种设备生产许可 | TSG 07—2019 + 《条例》第14条（图约束 C7） |
| 检验检测机构未取得核准 | TSG Z7001-2021 / Z7002—2022 |
| 许可到期后仍存续经营范围 | 时序约束 |

#### 专题二：直播电商

| 检测点 | 说明 |
|---|---|
| 经营范围含直播 / 网络文化经营类条目 | 新业态，**多数条目不在规范表述目录中** → 目录缺口高发区 |
| 批量个体户注册（同址 + 同代办 + 同时段） | MCN 机构批量注册特征 |
| 异地实际经营 | 注册地与经营特征地不符 |
| 名称 / 范围含 2026 禁用词 | "虚拟货币""稳定币""RWA" 等 |

> **直播电商是本方案的最佳 Demo 场景**：它同时集中了"新业态导致目录缺口""批量注册""规则滞后"三个赛题痛点。

## 9. 智能辅助应用（对应目标 3）

**总原则：LLM 严格旁路。它不产生任何新判断。**

### 9.1 智能问答（法规 RAG）

- 语料：dj.samr.gov.cn 政策专栏（数十条官方问答，每条带法规原文链接）+ 第一、二梯队法规全文
- **必须带条款级引用**。回答"能不能这么注册"要给出《条例》第几条
- 检索：向量召回 + 法规名/条号字面匹配混合
- **时效性**：依据 `law_anchor` 的生效/失效日期过滤，避免引用已废止条款（如 2022 年版文书规范已于 2026-05-01 废止）

### 9.2 智能问数（图谱 KBQA）

- 自然语言 → Cypher / SQL。走图谱查询而非纯 NL2SQL，更准也更好讲
- 示例："查一下江苏今年新设的直播电商类个体户里同址超过 10 家的"
- **只查标签库与图谱，不查原始数据**（保证口径一致）
- 返回结果附带生成的查询语句 → 可审计

### 9.3 智能报告

- 对标总局本来就在人工出的《数据质量情况报告》《数据分析报告》
- **每个结论都能点开看到支撑数据和命中规则** —— 这是它区别于"套模板"的唯一凭证
- 结构：宏观态势 → 中观专题 → 微观典型案例 → **规则缺口建议**

### 9.4 构造式解释与忠实性评测（本方案的学术性创新）

**错误做法（须明确避免）**："不仅输出风险分数，还自动解释风险依据"——这句话假设了**先有分数、LLM 事后编一个解释**。这叫 post-hoc rationalization，是 XAI 公认的坏做法，在监管场景中致命：**LLM 生成的解释可能与真实判定逻辑毫无关系**。

评委只需问一句"你怎么保证这个解释真的对应判定逻辑，而不是模型编的"——此题无解。

**正确做法（explanation-by-construction）**：判定本身即可解释（规则命中即证据链），LLM 不产生任何新判断，只把已有证据链**翻译成人话**。

**答辩杀器**：

> **"我们的系统不是先判断再解释，而是先有证据再有结论。解释不是 LLM 生成的，解释就是证据链本身，LLM 只做排版。"**

**再加一层——LLM 输出忠实性评测**：

| 评测维度 | 方法 |
|---|---|
| **事实忠实性** | LLM 输出中的每个事实断言，是否都能在 `EVIDENCE` 中找到对应？ |
| **法条忠实性** | 引用的条款是否真实存在？是否为当时有效版本？ |
| **无捏造率** | 输出中不可归因的断言占比（目标 < 1%） |
| **覆盖完整性** | 证据链中的关键事实是否被完整表述（无选择性遗漏） |

> 这是学术界的 hallucination / faithfulness 评测，落到监管场景就是"**AI 输出可审计**"。**可以负责任地判断：不会有第二个队伍做这个。**做出来放进文档一页，即为全场唯一一个"我们证明了 AI 没在胡说"的队伍。

## 10. 创新点（按优先级）

排序原则：**别人做不出来 × 我们做得出来 × 赛题真的要**。

### ① 规则即数据：可版本化的合规规则中台

七列规则表存库；规则引擎只做解释器；**改规则不发版**；**误报自动回流成目录缺口建议**。

- **直接回答赛题白纸黑字写的"规则更新滞后"**
- 这是**机制创新**而非技术创新 —— 但恰恰是评委（监管系统的人）最痛的点
- 没人做，因为所有队伍都在卷模型

### ② 跨省登记数据的图谱一致性检测

赛题原文写的是"**数据矛盾**"，不是"风险传播"。数据矛盾是**约束违反**：用形式化图约束检测，**零误报、100% 可解释**（见 §6.3 的八条约束）。

- "属地办理"必然导致跨省数据互相矛盾 —— 这正是"分散核验缺乏全国统一管控"的直接后果
- 别人做不了，是因为他们把"数据矛盾"当成异常检测去拟合

### ③ 构造式解释 + LLM 输出忠实性评测

在②基础上加一层：**检验 LLM 的解释是否严格来自证据链、有无捏造**（见 §9.4）。

### ④ 把监管方自己变成标签挂载对象

效能标签挂在**流程单据和登记机关**上：全省退回率排名、整改闭环时长分布、各省规则误报率对比。**"各地标准不一"由此从一句话变成一张图。**

### ⑤ 行为序列模式识别（而非"风险预测"）

**这里有一个必须讲清楚的边界**：

**"时序演化 + 提前预警"是雷，不能碰**：
- **业务上不成立**：登记注册是**一次性行为**，不是持续经营。"风险从形成到演变的全过程"这个叙事来自信贷风控，登记场景里没有这个过程 —— 设立就是那一天。
- **法律上危险**："提前预警"意味着预测。预测谁**将要**虚假申报？评委会问："人家还没做，你凭什么预警？法律依据呢？依据预测拒绝企业开办，谁担责？"

**真正的时序维度是识别已发生的行为序列**：

> 设立后 7 天内变更法定代表人 → 30 天内变更全部股东 → 60 天内变更住所

这是"壳公司过户"的经典链路，**是已经发生的事实序列，不是预测**。同类：代理人代办量从 5 笔/月突跳到 200 笔/月；某地址 30 天内新增 47 家。

> **监测已发生的行为序列 ≠ 预测未来。**前者合法、可解释、可追责；后者是雷。**在文档中主动写明这个边界，评委会认为团队懂监管。**

### 关于常见"伪创新"的说明

| 常见提法 | 为什么不能作为创新点 |
|---|---|
| "多维标签体系（主体/行为/关系/风险）" | **是赛题原文**。把题目当创新点，等于告诉评委在复述题干。且四分法是数据中台 PPT 标配，"标签组合而非单标签"就是规则引擎的 AND/OR |
| "关系图谱 + 风险传播分析" | 方向对但落后十年。职业法人、批量注册识别早已被商业征信产品化，各省信用风险分类系统也都有。评委中只要有一个懂行的，此条掉分 |

> ①②③ 有个共同点：**它们都不是"更准的模型"，而是"更可信的机制"**。这是本题的正确姿势。

## 11. 评测方案

| 层次 | 指标 | 数据 |
|---|---|---|
| **规则层** | 每条规则的命中率、误报率；规则覆盖的法条比例 | 合成数据 |
| **图约束层** | 约束违反检出率（理论应为 100%，因为是逻辑判定）；执行耗时 | 合成 + 真实 |
| **标签层** | 20 个核心风险标签的 P / R / F1；命中率分布；稳定性 | 合成数据 ground truth |
| **图谱层** | 实体对齐准确率（尤其**地址归一准确率**）；图谱完备性 | 真实 200–500 家 |
| **问答层** | 条款引用准确率；答案可归因率 | 政策语料（真实） |
| **忠实性层** | 无捏造率；事实可归因率；法条引用有效性 | 全部 |
| **业务层** | **日均告警量 / 复核人力比**；阻断级标签 precision（须 > 0.99） | 合成数据外推 |

**主动报告的负面结果**（学术诚信，也是加分项）：
- 哪些规则命中率为 0，为什么
- 哪些标签在噪声下崩溃
- 地址归一的失败案例类型
- 告警负荷的估算及其对人力的要求

## 12. 技术实现与选型

| 层 | 选型 | 理由 |
|---|---|---|
| 图数据库 | Neo4j（社区版）/ NebulaGraph | Cypher 表达力足够写图约束 |
| 约束引擎 | Cypher 约束查询 + SHACL（可选） | 形式化、零误报 |
| 规则引擎 | 自研表达式解释器 或 Drools | 规则存库、解释执行 |
| 向量库 | Milvus / pgvector | 法规切片检索 |
| 编排 | Dify（自托管） | 团队已有实践经验 |
| 后端 | FastAPI / Python | 与图算法、评测脚本同栈 |
| 前端 | Next.js + TypeScript + Tailwind | 团队已有实践经验 |
| 图可视化 | Cytoscape.js / G6 | 证据链子图展示 |
| 数据合成 | Python + Faker + 自研注入器 | 需产出 ground truth |

> **不要做的事**：登录页、用户管理、多角色权限。不加分，纯烧时间。Demo 只需 happy path。

## 13. Demo 设计与演示脚本

### 13.1 四屏结构

```
① 宏观驾驶舱（全国态势、数据质量得分）
        │
        ▼
② 中观专题（特种设备、直播电商）
        │
        ▼
③ 风险清单（按标签筛选、分级处置）
        │
        ▼
④ 单企业证据链  ◄── 全场最重要的一屏
   图谱子图 + 命中规则 + 法条锚点 + 处置建议

   智能助手（问答 / 问数 / 报告）贯穿全程
```

**第 ④ 屏必须打磨到极致**，因为它一屏同时展示了：图谱、规则引擎、法条溯源、可解释性、处置建议。评委看完这一屏，技术含量就传达完毕。前三屏只是把他领到这里。

> **关键：Demo 要能从驾驶舱一路点到某一家具体企业的证据链。这条下钻路径本身，就是"宏观-中观-微观立体化分析体系"的现场证明。**评委不会读文档，但会看你点这一路。

### 13.2 演示主线：一条数据的完整旅程

**输入**：江苏某个体工商户设立申请。经营范围含"网络直播服务"，住所"XX路88号2001室"，经营者李某，经办人为代理人张某。

| 步骤 | 动作 |
|---|---|
| **1. 治理层** | 统一社会信用代码校验 → 地址标准化到门牌粒度（"88号2001室" → 标准 address_id）→ 代理人张某对齐到全国代理人库唯一 ID |
| **2. 知识层** | 落图。生成 5 类节点：企业、经营者、地址、代理人、经营范围条目，及其间的边 |
| **3. 计算层（三路并行）** | **规则引擎**："网络直播服务"是否在《经营范围规范表述目录》？→ **不在** → 命中 `表述不规范`（规范性，仅打标）<br>**图算法**：该 address_id 已关联 213 家主体，近 30 天新增 47 家 → 命中 `住所批量聚集`（统计异常，人工复核）<br>**图算法**：代理人张某近 30 天代办 156 笔、核准率 41% → 命中 `代办行为异常`<br>**冲突检测**：经营者李某同时在 3 省任 4 家个体户经营者 → 命中 `身份异常`（需复核） |
| **4. 标签库** | 写入 4 条标签实例，各带 (挂载实体, 标签, 时点, 证据ID, 置信度) |
| **5. 输出** | **微观** → 工单"建议人工复核"，附证据链：图谱子图 + 4 条命中规则 + 各自法条锚点<br>**中观** → 该地址进入"重点关注集群"清单<br>**宏观** → 江苏·直播电商类新设主体·地址聚集指数 +0.3 |
| **6. 回流** | 复核员判定"3 条成立、1 条误报——`表述不规范` 其实是新兴业态，目录里根本没有这个词" |

**第 6 步那个误报，是整个系统最值钱的输出。**

它不是 bug，它自动生成了一条建议：

> **"《经营范围规范表述目录》存在缺口，建议上报总局补充'网络直播服务'条目"**

而这恰恰是政策明文规定的动作 —— 地方应当及时向总局反馈新兴行业经营项目，由总局定期更新维护规范目录。

> **本系统把"人工发现、层层上报"变成了"误报自动沉淀为目录更新建议"。**
>
> 这条路径讲完，评委就明白：这不是一个报警器，而是**一个监管规则的自我更新机制**。这是本课题真正的天花板。

## 14范围控制（重要）

- **不做全国全量，做「一个省 × 两个专题」的纵深。**3 人 + 无数据，铺开必死。评委宁可看一个做透的江苏，不想看一个假的全国。
- **60–80 个标签，不是 500 个。**
- **不做登录、权限、用户管理。**

## 15. 风险与答辩预案

| 评委必问 | 标准答案 |
|---|---|
| **"你的误报怎么办？"** | 分级处置：高置信法定强制规则自动拦截（precision > 0.99 才允许设为阻断）；中置信人工复核；低置信仅打标。且**误报会回流成为规则调优的标注数据**。承认告警疲劳并给方案，比吹准确率有说服力 |
| **"规则更新滞后怎么解决？"** | **规则可配置化 + 法规变更触发规则库版本更新 + 误报自动沉淀为目录缺口建议**。不是"重训模型" |
| **"总局和地方标准不一，你怎么统一？"** | **统一规则中台 + 属地差异化参数**。不说"全国一刀切"，会被业务评委打 |
| **"数据是假的，结论有意义吗？"** | 数据是合成的，但**规则、流程、法条引用全部真实可溯源**；且正因合成，我们才有 ground truth，能报 P/R —— 用真实数据的队伍反而做不到 |
| **"你们给企业评分是否合法？"** | 引《关于推进企业信用风险分类管理的意见》：分类结果是**监管资源配置的内部参考依据，不是对企业的信用评价**，不对外、不公示 |
| **"LLM 会不会误判导致企业开不了业？"** | **LLM 不参与任何判定**，严格旁路。判定由确定性规则引擎产生，LLM 只做证据链的翻译与排版 |
| **"你怎么保证 LLM 的解释是真的？"** | explanation-by-construction + 忠实性评测（无捏造率、事实可归因率、法条引用有效性）。**"解释不是 LLM 生成的，解释就是证据链本身。"** |
| **"一天报多少条？监管人员看得完吗？"** | 已做告警负荷估算：日均 X 条 / 复核人力 Y 人 = 人均 Z 条。分级后阻断级仅占 x% |
| **"和天眼查/企查查有什么区别？"** | 商业征信做的是**事后关系挖掘**；本系统做的是**事前合规校验 + 数据矛盾检测 + 规则治理闭环**，且每条结论挂法条锚点。商业产品没有法源，也无法回传整改 |

### 常见死法（自查清单）

- [ ] 把"多维标签体系"当创新点 → 那是赛题原文
- [ ] 标签生成排在知识图谱之前 → 关系标签全废
- [ ] LLM 站在决策主链上 → 无法回答担责问题
- [ ] 只做企业标签，没有代理人 / 单据 / 登记机关标签 → 丢掉"效能短板"
- [ ] 讲"提前预警"/"风险演化" → 法律上答不出
- [ ] 答辩里花时间讲 AUC → 跑偏；监管场景关心的是"凭什么"和"错了怎么办"
- [ ] 做了登录页和权限系统 → 白烧一周
- [ ] 智能报告是套模板 → 一眼看穿；必须每个结论可点开溯源
