# 项目整体梳理

## 1. 项目一句话

本项目不是用大模型判断企业是否违法，而是把登记法规和监管口径转成可版本化规则，从登记数据中生成可追溯的异常线索、标签、证据和分析指标，供监管人员复核、研判和促进规则更新。

## 2. 项目真正要回答的三个问题

1. **依据问题**：登记业务中哪些内容应该统一校验，法规、属地实现和数据字段之间存在哪些断点？
2. **计算问题**：如何把上述要求转成可执行、可追溯、可调整的规则和数据标签？
3. **使用问题**：如何让人员能快速看到态势、下钻证据、查询法规并生成报告，同时不让 LLM 替代监管判断？

## 3. 三个赛题目标的正确关系

```text
目标1：定标准和数据需求
  法规 / 政策 / 9省申报矩阵 / 两级流程 / 堵点
                         ↓
目标2：建设核心监测资产
  数据治理 → 规则引擎 → 标签+证据 → 宏观/中观/微观分析
                         ↓
目标3：提供使用入口
  法规问答 / 受限问数 / 证据驱动报告
                         ↓
人工复核与整改结果回流
  促进阈值、规则、目录和政策口径更新
```

其中：

- 目标1是**研究与业务基础**，不是监管系统的日常操作页面。
- 目标2是**项目核心产品**，前端主要应展示这一层的使用过程。
- 目标3是**目标2的自然语言入口**，不产生新数据和新风险判定。

## 4. 系统业务主链

```text
属地申报/核准数据
  → 归集与实体对齐
  → 法定规则 / 统计异常 / 图关系 / 专题规则
  → 标签实例 + 证据快照
  → 宏观态势 / 中观专题 / 微观主体画像
  → 人工复核
  → 成立则进入整改工单，误报则进入规则治理
  → 复核/整改结果回流
```

系统“发现问题”的含义是生成可复核线索，不等于自动认定企业违法。

## 5. 当前已有工程产物

| 层次 | 已有产物 | 当前边界 |
|---|---|---|
| 研究基础 | 9省申报矩阵、核心法条、七列规则表、正向与整改流程 | 整改角色、时限和字段仍需真实业务访谈验证 |
| 数据模型 | 23张表、2个视图，覆盖企业、人员、地址、业务单据、登记机关和登记代理人 | 当前为合成数据，真实地址标准化尚未验证 |
| 规则与标签 | 20条规则、20个标签、227条当前标签、227条证据 | 不能用合成数据声称真实业务准确率 |
| 复核回流 | 227条模拟复核：215条成立、12条误报；支持实测precision和阈值校准 | 复核结果由合成ground truth生成，不是真实工作人员判定 |
| 三级分析 | 10项指标、122条指标实例；直播电商专题14家/5家许可缺失；29个关联社区/160家成员；单企业画像视图 | 特种设备专题和跨省图规则尚未完成 |
| 数据服务 | FastAPI指标、标签、企业画像、集群与结构化报告接口 | 前端尚未全面改为接口实时读取 |
| 智能辅助 | 前端交互样例与可供Dify调用的数据接口 | 法规RAG、自然语言编排和忠实性评测未完成 |
| 整改办理 | 权责边界、状态机及两条可交互前端样例（企业数据整改 / 规则误报治理） | 没有真实工单表、持久化、账号权限、消息通知和业务接口 |

## 6. 前端应该如何组织

前端面向“总局/省级监测研判人员”，导航不再按三个赛题目标分组，而按人员的工作顺序组织：

1. **监测总览**：数据范围、当前资产、主链状态和重点发现。
2. **线索清单**：按法定强制、统计异常、规范性和情报线索分类查看。
3. **证据核验**：查看命中快照、法条/基线、规则版本、处置边界和复核结果。
4. **三级研判**：宏观趋势、中观专题/关联社区、微观企业画像。
5. **整改闭环**：如实展示为“已设计、未开发”，后续承接成立线索。
6. **规则治理**：查看规则类型、处置级别、命中数、复核准确率和阈值校准。
7. **智能辅助**：仅作为问法、问数和报告入口，明确标注未接入的部分。
8. **项目说明**：专门说明三个目标的对应关系和当前完成程度。

## 7. 演示时只走一条主路径

```text
监测总览
  → 点开“住所批量聚集”线索
  → 查看 P99基线、30日窗口、字段快照和人工复核结果
  → 回到中观住所/直播电商专题
  → 展示误报如何进入规则校准
  → 演示成立线索进入企业数据整改、误报进入规则治理；明确真实后台仍是下一阶段功能
  → 最后用智能助手查询同一个指标/法规
```

这条路径同时证明目标2是核心、目标1是依据、目标3是入口。

## 8. 汇报页的正确结构

1. 赛题痛点与项目边界。
2. 三个目标如何前后衔接。
3. 总局端监测机制，以及“线索≠违法认定”。
4. 一个具体线索如何从原始数据变成证据。
5. 现有工程资产和三级分析实物。
6. 已经跑通的技术链与尚未实现的整改链。
7. 没有真实数据时的验证方法与诚实边界。
8. 下一阶段的三个收口任务。

## 9. 当前对外表述

> 本项目已用合成数据跑通“数据建模—规则计算—标签证据—三级研判—复核回流”的技术骨架，并完成总局—地方整改的双案例前端状态机；真实数据接入、法规RAG、前后端联调和整改工单后台尚未完成。
