# 规则表、数据标签与跨省申报矩阵说明

## 1. 目前做了什么，分别服务什么

| 已有资产 | 作用 | 服务环节 |
|---|---|---|
| 跨省登记申报矩阵 | 对比各省申请材料、申报字段和办理差异 | 确定全国统一字段、属地差异参数和数据缺口 |
| 七列规则表 | 把法规政策翻译成明确的监管判断 | 建立统一监管标准，发现无法自动判断的监管堵点 |
| 规则库 `RULE` | 保存可执行逻辑、触发时点、处置方式和版本 | 由规则引擎统一执行，规则调整不修改引擎代码 |
| 标签字典 `TAG_DICT` | 定义标签名称、挂载对象、依据、更新频率和失效条件 | 统一系统输出口径 |
| 标签实例 `TAG_INSTANCE` | 记录某个对象在某个时间命中了什么标签 | 支撑预警清单、企业画像和历史追溯 |
| 证据表 `EVIDENCE` | 保存命中时的字段快照、规则版本和法规依据 | 回答“为什么预警”，支撑人工核查和整改 |
| 指标库 | 对登记数据和标签进行分组、聚合 | 支撑宏观趋势、中观专题和登记效能分析 |

当前工程台账：**20条在库规则、20个标签定义、227条标签实例、227条对应证据、10项分析指标，标签挂载对象覆盖6类**。

整体关系：

```text
跨省申报矩阵确定数据口径
        ↓
七列规则表形成监管口径
        ↓
规则库执行判定
        ↓
生成标签实例和证据
        ↓
形成监测预警、三级研判和整改工单
```

## 2. 跨省登记申报矩阵是什么意思

文件：[10省登记申报矩阵.xlsx](./10省登记申报矩阵.xlsx)

它不是风险规则表，而是**跨省业务差异对照表**，主要回答三个问题：

1. 哪些申请材料和字段是各省普遍采集的，可以进入全国统一数据模型；
2. 哪些要求存在属地差异，应作为地方参数配置，不能全国一刀切；
3. 哪些信息线上系统没有采集，导致规则无法自动计算，需要接口补充或人工核查。

矩阵的直接产出：

- 已归纳出各省共同使用的核心字段，作为数据建模依据；
- 发现住所材料、名称条件等属地差异，作为“全国统一逻辑＋属地参数”的依据；
- 发现申报表看不到的信息不一定未采集，也可能属于独立备案模块，防止把“矩阵空白”误判为“监管缺失”。

### 当前数量口径

文件名保留为“10省登记申报矩阵”，但当前工程记录显示：**已完成并核实的是9省，材料级9/9、字段级9/9**。在第10省补齐前，汇报时建议称“跨省登记申报矩阵”或“9省已核矩阵”，不要直接声称10省均已完成。

## 3. 七列规则表是什么意思

文件：[七列规则表.xlsx](./七列规则表.xlsx)

七列规则表是**法规政策到系统规则的翻译表**。每一行表示一个可以核查的监管事项。

| 列 | 含义 | 解决的问题 |
|---|---|---|
| 法条锚点 | 规则对应的法规、条款或统计依据 | 规则依据是什么 |
| 字段 | 判断所需的数据字段 | 系统需要采集什么数据 |
| 判定逻辑 | 字段满足什么条件时命中 | 机器怎样发现问题 |
| 风险标签 | 命中后输出什么可读结果 | 系统如何描述问题 |
| 规则类型 | 法定强制、规范性、统计异常或情报线索 | 该结论有多强，能否直接处置 |
| 触发时点 | 事前、事中或事后 | 在登记流程哪个环节执行 |
| 处置级别 | 硬阻断、人工复核或仅打标 | 命中以后怎么办 |

七列规则表的两个重要作用：

- **形成统一标准**：相同法规要求采用相同字段、逻辑、证据和处置口径；
- **发现监管堵点**：如果“判定逻辑”写不出来，通常说明数据拿不到、事项依赖人工裁量，或者该要求不适合自动校验。

七列规则表是研究侧资产；数据库中的 `RULE` 是从中筛选并工程化后的可执行规则。二者不能混为一张表。

## 4. 规则是怎么得到的

规则形成步骤：

1. 从法规政策中提取“应当、不得、必须”等明确要求；
2. 记录准确的法规名称和条款，形成法条锚点；
3. 找出完成判断所需的登记字段；
4. 把文字要求写成等式、条件、时间差、目录匹配或关系查询；
5. 确定命中后输出的标签；
6. 区分法定强制、统计异常、规范性和情报线索；
7. 根据执行对象确定触发时点和处置级别；
8. 将可执行部分写入规则库，由统一引擎运行并保存版本。

示例：

```text
《公司法》第47条
→ 需要注册资本、股东认缴额字段
→ SUM(股东认缴额) ≠ 注册资本
→ 生成“出资与注册资本不符”标签
→ 事后发现时进入人工复核和整改
```

## 5. 数据标签是干什么的

标签是规则执行后的**结构化结果**，表示某个监管对象命中了一个可复核事实。

标签主要用于：

- 生成预警线索；
- 汇总单个企业的风险画像；
- 聚合形成地区、专题、登记机关指标；
- 关联证据和法规依据；
- 记录标签何时生效、何时失效；
- 在人工复核后进入企业整改或规则治理。

需要区分三个概念：

| 概念 | 示例 |
|---|---|
| 特征 | 同址主体数 = 50 |
| 标签 | 该地址命中“住所批量聚集” |
| 指标 | 全省聚集地址占比 |

标签不是风险分，也不是违法认定。统计异常标签必须经过人工核查。

## 6. 标签是怎么得到的

标签体系来自两条路径：

1. **自下而上**：对七列规则表的“风险标签”列去重，形成候选标签；
2. **自上而下**：根据宏观、中观、微观分析需要，补充专题、效能和关系类标签。

候选标签还要补齐以下元数据后才能进入标签字典：

- 挂载对象；
- 标签值类型；
- 法规或统计依据；
- 命中后由谁处理、如何处理；
- 更新频率；
- 失效条件；
- 标签版本。

规则引擎运行后，才会产生具体的 `TAG_INSTANCE`；同时生成 `EVIDENCE`，保存命中时的字段和规则版本。

## 7. 什么是标签挂载对象

“挂载对象”就是**这条标签描述的是谁或什么**。标签不能全部挂在企业上，否则无法表达人员风险、地址风险、办理效能和事前申请问题。

| 挂载对象 | 标签示例 | 为什么挂这里 |
|---|---|---|
| 企业 | 出资与注册资本不符、许可项无批准文件、应注销未注销 | 问题属于已登记企业本身 |
| 自然人 | 职业法人、失信人员违规任职 | 问题属于人员身份或任职行为 |
| 地址 | 住所批量聚集 | 聚集现象由同一地址关联多家企业产生 |
| 登记代理人 | 代理人未备案、代理人兼任登记联络员 | 问题属于代理行为，不属于被代理企业 |
| 业务单据 | 申报出资不符、名称缺失、变更登记逾期 | 事前阶段企业可能尚未成立，只能挂在申请件上 |
| 登记机关 | 退回率异常偏高、平均办理时间异常 | 描述的是办理尺度和监管效能 |

当前标签体系已覆盖上述6类挂载对象。

## 8. 一句话区分

- 跨省申报矩阵：确定各地采什么、哪里相同、哪里不同；
- 七列规则表：确定依据什么、怎么判断、何时执行、如何处置；
- 规则库：让系统执行判断；
- 标签库：保存判断结果；
- 证据表：解释为什么得到这个结果；
- 指标库：把结果聚合成宏观和中观分析。
