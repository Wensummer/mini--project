# 项目汇报逻辑

## 一、汇报主线

本项目解决的核心问题是：各地企业登记申报和校验口径存在差异，总局难以按照统一口径发现登记数据中的矛盾和异常；问题发现后，缺少标准化的下发、核查、回传、复核和销号机制。

因此，项目不是单独建设风险模型或数据看板，而是形成一条完整业务链：

```text
跨省调研与法规梳理
→ 建立统一监管标准
→ 按标准执行规则
→ 生成标签和证据
→ 形成宏观、中观、微观研判
→ 触发预警
→ 属地核查整改
→ 总局复核重算
→ 销号或优化规则
```

三个项目目标分别对应这条链的不同阶段：

- 目标1解决“依据什么标准监管”；
- 目标2解决“如何发现、分析和处理问题”；
- 目标3解决“监管人员如何方便地使用标准和研判结果”。

## 二、目标1：形成统一监管标准

### 1. 做了什么

目标1目前形成了四类基础成果：

1. 法规政策与监管要求梳理；
2. 跨省登记申报矩阵；
3. 七列规则表；
4. 地方申报、总局监测、属地整改回传流程。

跨省登记申报矩阵用于区分：

- 各地共同采集、可以全国统一的字段；
- 存在地方差异、应允许属地配置的参数；
- 当前数据不能支持、必须补充接口或人工核查的事项。

七列规则表进一步把法规要求转化为：

```text
法条锚点＋所需字段＋判定逻辑＋风险标签
＋规则类型＋触发时点＋处置级别
```

最终形成的统一监管标准，不是一份法规清单，也不是一条SQL，而是一个完整标准单元：

| 标准内容 | 回答的问题 |
|---|---|
| 监管事项 | 要监管什么 |
| 法规或统计依据 | 凭什么监管 |
| 适用范围 | 对哪些主体和业务生效 |
| 所需字段 | 需要哪些数据 |
| 判定逻辑 | 系统如何发现问题 |
| 触发时点 | 事前、事中还是事后执行 |
| 证据要求 | 预警时必须保留什么 |
| 处置方式 | 阻断、人工复核还是仅提示 |
| 全国统一部分 | 哪些内容地方不能修改 |
| 属地配置部分 | 哪些参数可根据地方依据调整 |
| 版本信息 | 标准何时生效、如何追溯 |

### 2. 统一监管标准在哪里看

目前可以从三个层次查看：

#### 研究侧

- [跨省登记申报矩阵](./10省登记申报矩阵.xlsx)：说明各地采集什么、哪里一致、哪里不同；
- [七列规则表](./七列规则表.xlsx)：说明法规要求如何转成字段、逻辑、标签和处置。

#### 系统侧

进入前端顶部导航的“标准规则”，可以查看统一监管标准中心。目前展示5项完整标准样板：

- 登记前许可项目核验；
- 注册资本与股东认缴一致性；
- 住所短期批量聚集监测；
- 登记代理人身份备案核验；
- 登记事项变更时限监测。

每项标准都展示法规依据、适用范围、字段、逻辑、触发时点、证据、处置方式，以及全国统一部分和属地可配置部分。

系统地址：[登记智鉴](https://dengji-zhijian-monitor.limatrix223.chatgpt.site)

#### 工程侧

- `RULE`：保存执行逻辑、规则类型、触发时点、处置级别和规则版本；
- `rule_param_override`：保存有依据的省、市、登记机关差异参数；
- `TAG_DICT`：保存规则可能产生的标签及其挂载对象；
- `EVIDENCE`：保存命中时的字段快照、规则版本和依据。

当前系统中的5项标准是完整标准样板；在库规则数量多于完整标准卡片数量，不能把“20条执行规则”等同于“已经完成20项统一监管标准”。后续需要继续为其余规则补齐适用范围、证据要求和全国/属地边界。

## 三、目标2：构建标签、分析和预警体系

### 1. 做了什么

目标2已经形成以下链路：

```text
登记数据
→ 规则执行
→ 标签实例
→ 证据快照
→ 分析指标
→ 三级研判
→ 预警线索
→ 整改工单
```

当前工程台账包括：

- 20条在库规则；
- 20个标签定义；
- 227条标签实例；
- 227条对应证据；
- 10项分析指标；
- 企业、自然人、地址、登记代理人、业务单据、登记机关6类挂载对象。

### 2. 预警是怎么得到的

预警不是人工录入，也不是大模型直接判断，而是从规则和指标中产生。

#### 第一类：确定性规则预警

系统根据法规要求检查字段关系。

示例：注册资本一致性预警。

```text
注册资本 = 100万元
股东认缴合计 = 148万元
→ 两者不一致
→ 命中 R-CZ-01
→ 生成“出资与注册资本不符”标签
→ 保存注册资本、逐股东认缴明细、差额和规则版本
→ 形成待人工复核预警
```

这类预警有法条依据，但系统仍只输出可复核事实，不直接代替人工作出最终处理决定。

#### 第二类：统计异常预警

系统先用历史数据或地区数据形成统计基线，再判断当前对象是否明显偏离基线。

示例：住所短期批量聚集预警。

```text
同址主体 = 50户
近30日新增 = 50户
全省同址主体数P99 = 12户
当前新增阈值 = 10户
且未命中合法集群地址白名单
→ 命中 R-ZS-02
→ 生成“住所批量聚集”标签
→ 形成待属地核查预警
```

统计异常只说明“值得核查”，不能直接认定虚假登记。属地必须排除合法集群登记、集中迁入和地址标准化错误。

#### 第三类：专题交叉预警

先用专题标签圈定重点群体，再与其他合规标签交叉分析。

示例：

```text
经营范围包含直播电商相关关键词
→ 进入直播电商专题
→ 再检查许可批准文件、住所聚集和经营范围规范性
→ 形成专题指标和具体企业线索
```

直播电商专题标签本身不是风险结论，只有专题内进一步命中许可、住所等规则时才形成具体预警。

### 3. 预警等级怎么确定

预警等级应由以下因素共同确定：

- 规则类型；
- 法规效力；
- 异常程度；
- 是否已有人工复核；
- 对登记业务的影响；
- 误报造成的处置代价。

当前系统采用的展示口径：

| 等级 | 含义 | 处置方式 |
|---|---|---|
| 蓝色关注 | 态势或专题变化 | 继续观察和分析 |
| 黄色预警 | 统计异常 | 属地人工核查 |
| 橙色预警 | 法定字段或关系存在明显矛盾 | 重点人工复核 |
| 红色预警 | 人工确认且需要紧急处理 | 进入优先整改 |

颜色表示处置优先级，不表示已经完成违法认定。

### 4. 预警怎么展示

预警不能只展示一个数量或风险分，应展示完整的下钻链路。

#### 宏观层

展示省域、时间和总体变化，回答“发生了什么变化”。

例如：住所聚集线索近期上升、注册资本一致性标签集中出现。

#### 中观层

定位地区、登记机关、重点专题或关联群体，回答“问题集中在哪里”。

例如：定位到南京市栖霞区、某登记机关或某个聚集地址。

#### 微观层

展示具体企业、人员、地址或申请件，回答“具体是谁、为什么预警”。

至少包含：

- 预警编号和等级；
- 监测对象；
- 命中标签；
- 规则编号；
- 字段快照；
- 法规或统计依据；
- 当前状态；
- 建议处置方式。

系统中的展示路径是：

```text
监管总览
→ 监测预警
→ 宏观发现
→ 中观定位
→ 微观核验
→ 查看字段证据
→ 进入对应整改工单
```

### 5. 预警如何处理

人工核查后分为两条反馈路径。

#### 企业问题成立

```text
地方核查原始档案
→ 确认问题
→ 地方更正数据或要求补正
→ 上传整改前后证据
→ 总局复核
→ 规则重算不再命中
→ 销号
```

#### 规则误报

```text
地方核查证明企业或地址符合规定
→ 企业数据不修改
→ 回传政策依据和核查材料
→ 调整有依据的白名单、阈值或规则参数
→ 重新计算
→ 销号并保留误报记录
```

因此，目标2最终形成的不是一个风险看板，而是：

> 规则发现问题，标签描述问题，证据解释问题，三级研判定位问题，整改闭环解决问题，复核结果继续优化规则。

## 四、目标3：智能辅助使用

智能助手是目标1和目标2的自然语言入口：

- 问法规：查询统一监管标准及法规依据；
- 问数据：查询已经计算好的指标；
- 问预警：查询标签、证据和处置状态；
- 问工单：查询整改进度和回传结果。

智能助手不直接生成监管规则，不绕过规则引擎重新计算风险，也不代替人工完成最终认定。

## 五、汇报结论

本项目已经把法规政策和跨省业务差异转化为统一监管标准，并进一步转化为可执行规则。规则运行后生成标签和证据，通过宏观、中观、微观三级研判形成预警，再通过总局下发、属地核查、整改回传、复核重算完成闭环。智能助手负责让监管人员能够用自然语言查询标准、预警和工单。

项目的核心成果不是单个模型或单个页面，而是：

```text
统一监管标准
＋ 数据标签与证据
＋ 三级分析预警
＋ 整改反馈闭环
＋ 智能使用入口
```
